# Plan: Installing & Integrating ADLC into OpenCode

**Status:** Proposal · **Branch:** `opencode-integration` · **Date:** 2026-06-18

---

## 1. Problem Statement

Today, the `@adlc/*` toolkit ships as 19 independent, gate-shaped CLI packages, managed under one `@adlc/cli` dispatcher (with the `@adlc/runner` phase engine pending merge from the sibling `feat/codex-integration-codex` branch). While highly composable, using the Agentic Development Lifecycle (ADLC) in a raw terminal environment requires developers to:
1. Know exactly which CLI tool maps to which lifecycle phase (P0–P7) or control dial (D1–D3).
2. Manually execute the appropriate tool at the correct moment.
3. Manage exit codes, environment files, and LLM API keys for model-backed gates.

ADLC dictates that **enforcement must live at the tool layer, not the prompt layer** (as detailed in [ADLC.md](file:///home/voodootikigod/Projects/voodootikigod/adlc/.worktrees/opencode-integration/ADLC.md): *"F5 routes around instructions; it cannot route around a hook"*). Relying on a human or an agent to remember to run a validation command defeats the lifecycle's structural safety guarantees.

**Goal:** Provide a native, deeply integrated OpenCode plugin that enforces the ADLC automatically at correct lifecycle boundaries. The model should discover the lifecycle natively, safety-critical gates must run automatically, and no external API keys are required for standard operations (with standard key fallbacks documented strictly for missing programmatic APIs).

---

## 2. Design Thesis: Natively In-Process via Bun

OpenCode is an open-source, terminal-first agent harness. It uses a client-server architecture with a Go-based TUI/CLI client and a **Bun/JavaScript backend**. This design provides a powerful exploit for our integration:

- **In-Process JavaScript Hook execution:** Unlike Claude Code (which relies on external JSON-configured shell scripts), OpenCode executes plugins **in-process** in its Bun environment. The ADLC plugin can run natively in JavaScript, invoking the `@adlc/cli` dispatcher or using Bun's fast `Bun.spawn`/`Bun.$` APIs to run gates, process JSON outputs, and reject tool use with minimal overhead.
- **Keyless Execution via `--prompt-only`:** All in-session, builder-facing LLM-backed gates in the ADLC toolkit (e.g., `spec-lint`, `premortem`, `parallax`, `consensus-fix`) support a `--prompt-only` flag. When triggered, the tool prints the prompt text to stdout and exits. The plugin intercepts this output, feeds it to OpenCode's active model client, gathers the response, and inputs it to the tool's stdin, enabling keyless operation. **The calibration tool (`review-calibration`) is a notable exception: due to its intensive multi-sample benchmark execution, it does not support `--prompt-only` and requires a configured LLM provider API key (e.g., ANTHROPIC_API_KEY) to run its benchmarks dynamically in the background.**
- **Model-Agnostic Adaptation:** Since OpenCode is model-agnostic (supporting local models via Ollama as well as remote models), the plugin must degrade gracefully or warn the user if a configured model lacks the capability to satisfy complex gates (e.g., a low-tier model running spec evaluation).
- **Proposed SDK Extension Interfaces:** The model-routing and isolated-prompting capabilities detailed in this plan require specific extensions to the OpenCode plugin API (e.g., dynamic session model overriding and isolated sub-contexts). These are documented herein as target plugin specifications, to be verified or implemented in the OpenCode plugin SDK.

## 3. Primitive Mapping (ADLC to OpenCode)

OpenCode's extensibility points (custom commands, JavaScript event hooks, and agents) align cleanly with the ADLC primitive mapping established in the multi-harness roadmap.

Following **Option D (separate, concern-focused bins)** from the Command Reconciliation ADR:
- `adlc <tool>` is the dispatcher (`@adlc/cli`) for executing individual gate tools.
- `adlc-runner <verb>` is the runner (`@adlc/runner`) for phase-evidence assertions (`adlc-runner run <phase>` and `adlc-runner accept --ticket <id>`).
  - **Proposed `@adlc/runner` Specification Extensions:**
    - `adlc-runner run p0` (triage assertion) and `adlc-runner run p4` (build-gate assertion) are extensions because triage runs outside the core p1–p7 lifecycle runner, and build assertion requires verifying compiled outputs and logs rather than static manifest evidence chains.
    - `adlc-runner upgrade --ticket <id> --key <path>` is an extension to securely record cryptographically signed triage upgrade events in the manifest.
    - `adlc-runner init-developer-keys` is an extension to initialize the developer's key pair locally during project bootstrap.
    - `adlc-runner rotate-developer-keys` is an extension to allow key rotation while preserving the historical chain of trust.

> [!IMPORTANT]
> The `@adlc/runner` package is a **pending-merge dependency** developed in the sibling `feat/codex-integration-codex` branch. Integrating it requires merging its binaries into the shared main branch. The proposed `@adlc/runner` specification extensions listed above are pending design merges and will be implemented within the runner package.

### 4.2. In-Process Hook Contract (Fail-Closed & Path Hardening)

To prevent runtime resolution failures, logic drift, and drifting copies (violating the ADR's companion direction), the plugin does **not** duplicate or manually vendor the core rails-checking, path-resolution, and patch-parsing helper functions. The file `plugin/adlc-opencode/rails-checker.ts` is a **thin, local harness adapter** that maps OpenCode-specific tool arguments to the canonical, shared validation function exported by `@adlc/core`, which is declared as a devDependency and **pre-bundled** at build time into the plugin's single-file distribution bundle (`plugin/adlc-opencode/dist/index.js`) using a bundler.

**Dependency Lifecycle & Rebuild Policy:** To address long-lived security vulnerabilities in the pre-bundled `@adlc/core` library:
1. **Version Pinning Policy:** The plugin `package.json` pins `@adlc/core` to a strict semantic version (e.g., `1.0.2` rather than wildcard or range formats) to guarantee dependency determinism.
2. **Automated Update Notification:** On `session.created` or `session.start`, the preflight command queries the npm registry (or checks local environment packages) asynchronously for new versions of `@adlc/core` or `@adlc/cli`. If a newer security release is found, it raises a TUI notification card advising the user to pull and re-install the latest version.
3. **Rebuild Guidance:** The adoption documentation instructs developers to rebuild the single-file bundle (`npm run build` or `bun build`) whenever a new `@adlc/core` patch is released or at least weekly, maintaining up-to-date security boundaries.

The hook executes with a strict fail-closed contract:
- **Base Tool and Event Names Caveat:** The base hook event signatures (e.g., `tool.execute.before`, `tool.execute.after`, `session.created`) and tool names (e.g., `write`, `edit`, `apply_patch`) are target API interfaces to be verified against the official OpenCode SDK specification during implementation.
- **Hook Gating Boundary & Subprocess Bypass:** The in-session hook strictly gates **agent edit-tools** (e.g., the intercepted file mutation tools). It does **not** intercept or block file system writes performed by the plugin's own CLI subprocesses (e.g., `Bun.spawn` running `adlc` or `adlc-runner` commands), which write directly via the Node `fs` API.
- **Phase Gating & Triage-Class Skips:**
  - If a ticket is active in the repository (`.adlc/tickets.json`), the hook enforces a **Phase Transition Gate** at the tool layer to prevent the builder agent from bypassing spec and decomposition checks.
  - **Triage Class Gating & Mechanical Override:** The ticket's triage classification is determined through an interactive human interview via `/adlc-ticket`. In headless, CI, or autonomous environments, the triage class is parsed from command arguments (e.g., executing the triage helper script `node .opencode/commands/triage.js --class Trivial|Bounded|Substantial|Architectural`) or the `ADLC_TRIAGE_CLASS` environment variable, defaulting to **Substantial** if unspecified. To prevent builder agents from reward-hacking the classification to bypass gates ([ADLC-Flaw-F5 / Mitigation M1]), the plugin enforces a **mechanical blast-radius check** as the primary unbypassable control. For Trivial tickets (which skip specification and decomposition and thus have no predefined file domains), they are strictly enforced to have a mechanical diff size limit (maximum of 3 files and 50 lines of code changes as configurable defaults in `.adlc/config.json`), cannot introduce any new files, and must target areas with pre-existing, load-bearing test coverage.
    - **Triage-Time Verification (Pre-Edit):** At triage time, the `/adlc-ticket` command checks if the target paths/files/functions to be modified have pre-existing coverage. To prevent developers from gaming the Trivial triage via hollow tests, the command runs a targeted dry-run test suite check (which may introduce latency depending on the suite size) to verify pre-existing test coverage of the target area. It performs a quick validation run on those existing tests (e.g., executing `hollow-test` on the target code block, verifying if they fail when the implementation is removed) to verify they are load-bearing, avoiding any Goodhartable coverage percentage checks (ADLC P3). If the existing tests are missing, fail to run, or are found to be hollow (passing even when implementation is removed), the ticket is automatically upgraded to **Substantial** in `.adlc/tickets.json` and the full linear gates are engaged.
    - **Assertion-Time Verification (Post-Edit):** During execution and at commit/assertion time (`adlc-runner run p0` or PR check), the active diff is evaluated. The verification check verifies that the actual changes do not exceed the mechanical limits (3 files/50 lines) and confirms that the modified files pass `hollow-test` and test suite assertions; if `hollow-test` fails or the test suite crashes, the verification fails, and the ticket is upgraded to **Substantial** in `.adlc/tickets.json`, blocking integration.
    - **Triage-Class Mid-Session Upgrade Consistency:** If a Trivial/Bounded ticket is upgraded mid-session to Substantial (either due to exceeding mechanical limits or failing post-edit coverage checks), the plugin appends a structured `triage_upgrade` event to `.adlc/manifest.jsonl`. To respect the single source of truth for dangerous logic, the cryptographic signature is generated strictly by `@adlc/runner` (invoked via `adlc-runner upgrade --ticket <id> --key ~/.config/opencode/keys/developer.key` by the plugin's `triage-upgrade.js` script) using a persistent developer private key stored outside the repository in OpenCode's persistent configuration vault (`~/.config/opencode/keys/developer.key`), rather than an ephemeral session key, to allow cross-session signature verification. The public key is recorded in `.adlc/config.json` at initialization.
      **Developer Key Security Controls:** To protect the confidentiality and integrity of this key:
      1. *File Permissions:* The key file must be restricted to owner-only read/write access (POSIX permissions `0600` or `0400`), and the keys directory `~/.config/opencode/keys/` must be set to `0700`.
      2. *Access Validation:* `@adlc/runner` and the plugin check permissions on startup, refusing to execute signed commands and failing closed if the permissions are wider than owner-only or if the file owner UID does not match the active shell execution context.
      3. *Key Generation:* The key pair (RSA-4096 or Ed25519) is generated locally by `adlc-runner init-developer-keys` during `/adlc-init` using Node's `crypto` module, with keys written directly using atomic filesystem writes.
      4. *Rotation & Compromise Procedures:* Key rotation is executed via `adlc-runner rotate-developer-keys`, which generates a new key pair, appends the new public key to `.adlc/config.json`, and signs a `key_rotation` transition event in the manifest using the old key to maintain the chain of trust. In the event of key compromise, the developer must regenerate the keys and execute `adlc gate-manifest rebuild --from-git-history` with multi-signature or admin override token approval.
      To prevent agents from fabricating this event to bypass P1+P2 gates, `gate-manifest verify` enforces strict cryptographic signature verification on all triage upgrade events using the recorded developer public key; any unsigned or unauthenticated upgrade records are treated as tamper violations, failing the manifest validation closed. This event marks all prior Trivial/Bounded P0 triage and auto-passed or skipped records as superseded and invalid. The ticket is then routed back to Phase 1 (Interrogate) and Phase 2 (Decompose) under the Substantial lifecycle schema. When `gate-manifest verify` validates the manifest chain, it detects the `triage_upgrade` event, ignores the superseded Trivial/Bounded records, and strictly enforces the full sequence of Substantial phase evidence (P1 spec audit, Human Gate 1 signature, P2 atomic decomposition, P3/P4 rail/build assertions, and P5 prosecution loops), ensuring manifest consistency and preventing bypass.
    To prevent the lifecycle from entering an undefined state where implementation changes exist before a specification is written, the hook immediately blocks further tool edits, and the plugin itself stashes the current implementation changes safely (using `git stash push -m "adlc_triage_upgrade_backup"` to prevent data loss) before mechanically executing the revert of the uncommitted changes (via `Bun.spawnSync(['git', 'restore', '--staged', '--worktree', '--', ...files])` or equivalent git commands scoped to the modified files to clear both the index and the working tree, protecting against staged rail bypasses) to clean the workspace, preventing a reward-hacking agent from bypassing the block. The ticket is then moved back to Phase 1 (Interrogate) and Phase 2 (Decompose) to establish proper specifications and atomic partitions before rebuilding. To prevent pickaxe query vulnerabilities in long-lived repositories (where ticket IDs recur or appear in comments), the triage command `/adlc-ticket` writes the current trunk base commit hash directly to `.adlc/tickets.json` as `triageCommit` at triage time. The runner `adlc-runner run p0` strictly asserts that: (1) a valid ticket metadata entry exists in `.adlc/tickets.json`, and (2) a dispatcher check verifies the active diff has not violated the blast radius restrictions for the declared triage class. While interactive setups allow the human developer to pick the initial classification as a best-effort configuration step, **no human sign-off signature or approval gate is recorded in the manifest at P0**, ensuring compliance with the two-mandatory-moments doctrine (ADLC.md Phase 0 triage guidelines).
  - If the active ticket is triaged as **Substantial** or **Architectural**, the hook intercepts and blocks *all* structured file edits to any repository path (returning `{allow:false}`) until `adlc-runner run p1` (spec interrogation) and `adlc-runner run p2` (decomposition) have successfully asserted.
  - If the active ticket is triaged as **Bounded**, the hook blocks all edits until the rails are written and frozen in Phase 3.
  - **P3-to-P4 Rail Freeze Transition:** When a Bounded/Substantial/Architectural ticket is active, the declared `rails` paths are initially marked as **Pending** (where edits are strictly restricted to the `rail-writer` context to defend creator/critic isolation, ADLC Principle 5). To ensure this isolation in environments lacking native agent-identity verification APIs (resolving the limitation that `hollow-test` cannot prove separate context), the hook **blocks all rail edits during builder sessions entirely**. Pending rails can only be written during a dedicated `/adlc-rail-write` workflow invocation (which establishes a secure temporary session token in `.adlc/tickets.json` and obtains a platform-agnostic file lock `.adlc/rail-write.lock` using `proper-lockfile`). Since tool calls are routed through the main OpenCode server process (rendering calling process PID verification within the hook infeasible), creator/critic isolation is enforced by having the plugin's slash command pipeline temporarily disable the builder agent session and update the active tool-execution rules to restrict tool edits exclusively to the rails directory while blocking edits to implementation paths. To prevent a fail-open state where a crash or force-quit during `/adlc-rail-write` leaves the temporal token active permanently, the preflight cleanup routine runs on both `session.created` and `session.start` (including session reconnect and resume events). The preflight hook verifies the status of the file lock; if the lock is stale (the file exists but no active process holds the lock), the hook automatically invalidates and clears the stale session token from `.adlc/tickets.json`, restoring the strict rail-guard posture. To prevent builder agents from writing to rails during this temporal window, the plugin TUI **suspends active builder subagent execution** for the duration of the `/adlc-rail-write` run, ensuring no builder tools execute concurrently. P3 is completed when `/adlc-rail-write` finishes execution (verifying the rails via `adlc hollow-test` and executing a compiler typecheck on the stubs and tests, and recording the rail-freeze in the manifest). Once P3 is asserted, the rails transition from **Pending** to **Frozen**, engaging the hook's absolute edit block on those paths for all agents (including the rail-writer).
  - **Trivial Gating & Hardening choice:** If the active ticket is triaged as **Trivial**, the hook skips P1, P2, and P3 phase transitions entirely (marking them as auto-passed/skipped in the manifest) and permits edits to non-rail paths immediately. For Bounded tickets, the runner similarly writes auto-passed/skipped records for P1 and P2 to `.adlc/manifest.jsonl` upon triage, ensuring `gate-manifest verify` has a complete evidence chain to validate. However, the PreToolUse hook still blocks Editor writes to frozen rail paths locally (not "zero" enforcement) to ensure historical rails remain protected. A prosecution pass (P5) running the full **5 lenses** (correctness, security, contract, diff, tests) as a single pass (complying with ADLC.md Phase 0 routing for Trivial tickets, where one prosecution pass is required rather than the full loop-until-dry protocol) runs automatically to verify basic safety.
    This prosecution is triggered by: (1) the developer running `/adlc-accept --ticket <id>` inside the active session (the primary happy path), which executes P5 prosecution, and upon success executes `behavior-diff compare` and records the Human Gate 2 signature in one combined step before session exit; (2) the plugin's `session.ended` / `session.terminated` close event; or (3) git commit attempts in the terminal (blocking via pre-commit hooks).
    **P5→P4 Fix Loop for Trivial Tickets:** If findings are detected during the Trivial prosecution pass, the plugin does **not** immediately scrap the changes. Instead, it prompts the builder agent in its active session with the structured findings checklist. The builder is granted a single fix loop round (up to $R_{\text{trivial\_fix}} = 2$ rounds total) to fix the implementation. If the second prosecution pass fails (Strike 2), or if the session is closed without resolution, the plugin then stashes the entire set of staged and unstaged changes (`git stash push -m "adlc_prosecution_upgrade_backup"` to preserve work), reverts the workspace (`git restore --staged --worktree -- .`), upgrades the triage class to Substantial, and routes the ticket to Phase 2 (Decompose) to establish spec and decomposition hygiene.
    **Out-of-Session Trivial P6 Integration Workflow:** If prosecution is triggered at session end or pre-commit, the session has terminated or is running in a terminal where slash commands are unavailable. To resolve this disconnect:
    - If prosecution passes at session close, the terminal or TUI prompts the developer: "ADLC: Prosecution passed. Record Human Gate 2 (behavioral acceptance) now? [Y/n]". If accepted, it runs `behavior-diff` and records the signature before terminating.
    - If the session closes or the pre-commit hook runs without a TUI, the developer can run the CLI command `adlc-runner accept --ticket <id>` in the terminal to review behavior-diff and record the P6 signature.
    - If the developer reconnects to a new OpenCode session, the preflight hook detects the pending P6 acceptance state and prompts the user with an action card to run `/adlc-accept` before any new tickets are processed.
    However, because local git hooks and session closing hooks are advisory or agent-deletable in a local-only environment (representing zero guaranteed mechanical enforcement), this local prosecution is considered best-effort and must be strictly backstopped by the CI/CD pipeline which runs the `adlc prosecute` check at PR/commit verification time, failing the build if prosecution is not dry. A deterministic build-gate check (verifying compilation and test run exit codes) always precedes the prosecution execution, and on success the manifest audit hook writes a P4 build record to `.adlc/manifest.jsonl` to avoid `adlc-runner run p4` failure paths at prosecution/assertion time. If the single prosecution pass succeeds, the plugin executes the runner assertion `adlc-runner run p5` as an automated side effect to write P5 phase evidence to the manifest. Conforming to the mandatory human gates constraint (ADLC.md Phase 0 routing diagram), Trivial tickets **must route through Phase 6 (Human Gate 2, behavioral acceptance)**. Once prosecution completes and behavioral acceptance is completed (via `/adlc-accept` or the out-of-session workflow), the human behavioral acceptance signature is recorded in the manifest before integration, maintaining the required gate signature.
  - If no ticket is active, the hook behaves as a **no-op** and immediately permits the tool call (no-op safety: clean repos cannot be bricked).
- **Suppression-Marker Gating (Goodhart Defense):** The rails-guard hook, pre-commit hook, and CI/CD workflows scan the diff (or parse file edits) and fail the check if any new test skip/xfail decorators (e.g., `test.skip`, `it.skip`, `describe.skip`, `it.only`, `describe.only`, `test.only`, `fdescribe`, `fit`, `xit`, `xdescribe`, `@pytest.mark.skip`, `@unittest.skip`, `@pytest.mark.xfail`, `@pytest.mark.skipif`) or lint-suppress comments (e.g., `eslint-disable`, `eslint-disable-next-line`, `ts-ignore`, `eslint-disable-line`, `// @ts-nocheck`, `/* tslint:disable */`, and `// biome-ignore` markers) are introduced, unless they are explicitly declared in the ticket's `allowedSuppressions` whitelist in `.adlc/tickets.json` (complying with ADLC C5 and the Goodhart Annex).
- **In-Session Direct Checks:** To avoid the performance overhead and nondeterministic failure risks of spawning shell commands or LLM calls on every tool use, the hook resolves and reads `.adlc/tickets.json` **directly via Node `fs`**.
- **Default-Intercept Path Filter & Unresolved Target Protection:** Instead of gating an explicit allowlist of tool names, the hook utilizes a **default-intercept** approach for structured tools: it intercepts *all* tool calls that accept path arguments, allowlisting safe read-only tools (`read`, `view`, `glob`, `search`, `grep`, or custom read-only tools declared in `.opencode/opencode.json` under `allowedReadTools` to prevent false-positive blocks on custom diagnostic or LSP tools), and inspects their target paths. If a tool call is write-capable (non-allowlisted) but its target paths cannot be resolved by the hook (e.g., unrecognized parameter names or arguments), the hook **must deny the execution (fail closed)** by returning `{ allow: false, reason: "Unresolved target path on mutation tool call" }` to prevent silent bypasses.
  - **Shell Tool Gating Exclusion:** Conforming to the decision to drop in-session Bash rail enforcement, shell execution tools (such as OpenCode's bash/command runner tools) are **explicitly excluded** from the unresolved-path fail-closed intercept rule. These tools are permitted to execute without in-session path verification, and all shell-based rail edits are caught at commit-time in CI/CD or the local pre-commit hook.
  - **Path Protection, Symlink & Hard Link Hardening:** The hook canonicalizes target paths. To prevent directory traversal and symlink bypasses on new file creation (where the file does not exist yet), the hook resolves symlinks on the *staged target path itself*, walking from the root directory segment-by-segment to verify that no path segment is a symlink resolving into a restricted directory. To prevent **hard link bypasses** (which point directly to the same file inode and are not resolved by `fs.realpathSync`), the hook compares the target path's device ID (`dev`) and inode number (`ino`) retrieved via `fs.statSync()` or `fs.lstatSync()` against the cached device and inode numbers of all active/frozen rail files. If a match is detected on any target path (even if the file name or folder path is outside restricted areas), the write operation is **blocked (denied)**. To mitigate TOCTOU (Time-of-Check-Time-of-Use) races in directory traversal, the hook and underlying file utilities use atomic write procedures (such as writing to a randomized temp file within the same directory and renaming it into place, or utilizing `O_NOFOLLOW` open flags where supported) rather than relying solely on the pre-check path resolution.
    At startup, the plugin resolves its own physical path using `fs.realpathSync(__filename)`. It locks this resolved canonical root path, protecting all files under the `.opencode/` directory from edits regardless of whether it is globally installed, locally copied, or symlinked.
    It specifically denies edits to:
      1. Declared `rails` paths (once frozen). To prevent subsequent builder sessions from tampering with historical tests after their target ticket is marked inactive, completed rails are appended to a global frozen rails index in `.adlc/config.json`. The hook enforces absolute edit protection on all historically frozen rails across all sessions, unless a new active ticket explicitly declares a rail-modification scope in its contract.
      2. The `.adlc/` directory (preventing editing of `tickets.json` or `manifest.jsonl`).
      3. The canonical plugin root path and the entire `.opencode/` directory, including the `plugin/`, `command/`, `agent/`, and `skill/` subdirectories and the `opencode.json` configuration file (preventing builder agents from altering plugin behavior, commands, skills, or agents in-session to bypass verification).
      4. The `.git/` directory (preventing deletion or manipulation of hooks, except during `/adlc-init` execution). Since `/adlc-init` runs directly as a plugin command via Bun backend `fs` operations, it writes to `.git/hooks/pre-commit` directly. The in-session hook strictly gates **agent edit-tools** and does not intercept plugin backend `fs` writes, so `/adlc-init` is allowed to configure the pre-commit hook without any bypass token.
      5. The `.github/` and CI config directories (preventing editing of workflows like `ci/rails-guard.yml`).
- **`apply_patch` Payload Parsing & Git Extended Headers:** The helper parses patch payloads to identify target files and evaluate them against protected paths. To close bypass vectors, the patch parser explicitly parses git extended headers (such as `rename from`, `rename to`, `copy from`, and `copy to`) in addition to unified diff markers (`+++ b/<path>`), ensuring that renaming or copying unrestricted files into restricted rail/configuration target paths is caught and blocked.
- **OpenCode Fail-Closed Hook Configuration:**
  - The plugin does **not** call `process.exit()` (which would terminate the in-process Bun backend and crash the entire OpenCode server). To ensure deterministic gating across all OpenCode versions (since thrown JS exceptions inside handlers may default to allow depending on harness implementation), the hook handler **never relies on throwing JS exceptions to block execution**; it strictly returns the structured native block payload: `{ allow: false, reason: "..." }`.
  - **onFailure Hook Posture (Proposed SDK Extension Specification):** Conforming to ADR Refinement 3, a fail-closed contract cannot live solely inside the plugin's JavaScript execution (since syntax or loading errors would prevent the JS from running, causing a fail-open default). Instead, the plugin registration in `.opencode/opencode.json` proposes a hook configuration option to fail-closed on loading errors or hook crashes:
    `{ "event": "tool.execute.before", "handler": "./plugin/adlc-opencode/dist/index.js", "onFailure": "deny" }`
    *Note: If the harness does not support a native `onFailure: deny` hook configuration, any plugin loading error, syntax exception, or startup crash will default-allow (fail open) the in-session hook. In such environments, the rails-guard safety guarantee relies entirely on the CI/CD rails-guard workflow backstop.*
- **Automated Manifest Side-Effects:**
  - Non-human evidence (such as test runs, `hollow-test` results, `flail-detector` logs, and prosecutor verdicts) is automatically captured and appended to `.adlc/manifest.jsonl` as an automated side effect of tool completion (via the `tool.execute.after` hook), rather than relying on manual slash command ceremonies. Human-gate approvals (G1 spec approval `/adlc-approve-spec` and G2 acceptance `/adlc-accept`) remain human-signed ceremonies.
- **Auditable Bypass:** Setting `ADLC_RAILS_BYPASS=1` is only permitted if the environment variable is validated through a secure human-in-the-loop prompt in the TUI (via the proposed plugin UI API). In non-interactive/headless runs (where no TUI is available), the bypass is **refused (fails closed)** and the refusal event (including timestamps, target paths, and session contexts) is logged directly as a `"refused_bypass"` audit record in `.adlc/manifest.jsonl`. If the manifest write fails or the manifest file is deleted/absent, the bypass is strictly refused (fails closed) and the plugin blocks further execution, preventing silent bypasses via manifest deletion.

### 4.3. Keyless LLM-Gate Dispatch (Two-Phase Bridge & Stdio JSON Cascade Protocol)

Except for `review-calibration` (which executes direct background provider calls requiring an API key), all builder-facing `@adlc/*` LLM-backed gates comply with the `--prompt-only` execution contract, printing the generated prompt to `stdout` and **exiting immediately with code 0** (rather than holding stdin open or waiting for interaction). This single-shot behavior works perfectly for standard single-prompt gates.

However, to support multi-sample (E1 sampling diversity) or multi-round operations (like `parallax`, `premortem`, and `consensus-fix`), the plan specifies a **Proposed `@adlc` CLI Specification Extension**:
- **Proposed `--prompt-session` Protocol:** Multi-round and multi-sample dispatcher/runner commands support a persistent stdio session mode via a new `--prompt-session` flag. In this mode, the process does **not** exit immediately; instead, it holds the process open, bubbling up prompt request JSON envelopes to `stdout`, and waits for completion responses to be supplied via `stdin`.
  - **Single-Shot Fallback:** For tools lacking `--prompt-session` support, the bridge falls back to standard `--prompt-only` mode, extracting and answering prompts sequentially in independent, single-shot CLI invocations. Note that sequential single-shot execution degrades parallel draw semantics (sequential drawing instead of parallel independence) and adds spawn overhead for multi-sample/consensus gates (e.g. `parallax`, `consensus-fix`). Under this fallback, sequential execution breaks the E1 independence assumption (highly correlated draws), reducing it to standard interrogation unless distinct model endpoints are rotated to preserve diversity. The system logs a warning advising that parallel verification is degraded.

To bridge these gates keylessly, the plugin bridge implements a **Two-Phase Stdio JSON Cascade Protocol**:

#### Bin Execution PATH Resolution:
To resolve the install resolution and unpublished runner package constraints, the plugin does NOT declare `@adlc/cli` or `@adlc/runner` as direct npm package dependencies (which would crash installation on the unpublished runner). Instead, the plugin expects these CLI tools to be installed globally by the user. At runtime, the bridge resolves the paths of the binaries by searching the user's environment `$PATH` (or executing them directly via `Bun.spawn` using the environment PATH) and falls back to searching standard global npm bin prefixes, verifying their presence and logging detailed user-friendly error guides if missing.

#### Phase 1: Prompt Extraction & Cascade Loop
1. **Spawn Extraction Process:** The bridge spawns the absolute command with prompt extraction flags: `adlc <toolName> <args> --prompt-session` (falling back to `--prompt-only` if not supported).
2. **Concurrently Stream and Parse Output:** The bridge captures the stdout of the child process. Under `--prompt-session`, the dispatcher tools and runner bubble up prompts wrapped in structured JSON envelopes:
   `{"type": "prompt_request", "id": "session-uuid", "sample": 0, "round": 1, "prompt": "..."}`
   *Note: Under standard `--prompt-only` single-shot fallback, the raw text printed to stdout is wrapped in a default single-round, single-sample JSON request envelope.*
3. **Parallel/Async Model Resolution:** The bridge intercepts these envelopes, extracts the prompt text, and executes them concurrently using OpenCode's active model client in transient, isolated conversation sub-contexts:
   `const completion = await context.client.prompt({ message: promptText, isolated: true });`
   If the tool requests a specific model configuration, the bridge passes the target model endpoint in the prompt request. To prevent prompt-injection attacks where the model output contains nested JSON control envelopes, the bridge performs strict escaping, sanitization, and schema validation. The bridge parses the model's text completion, validates that it conforms to expected plain text or structural formats, and encodes the sanitized value inside the `response` property of the JSON envelope before writing to the child process's stdin. To prevent control signal injection, the bridge does not use fragile regex-based detection to escape JSON-like patterns; instead, it uses the canonical `JSON.stringify()` serialization mechanism to safely wrap the model's text response as a JSON string literal inside the stdin envelope, ensuring all nested braces, quotes, and escape characters are parsed strictly as inert values by the child process.
   **Programmatic Client Fallback / Manual Bypass:** If OpenCode's programmatic prompting client API is missing, disabled, or refuses to execute, the bridge automatically degrades to a **Bypass-to-User Prompting Mode**: it outputs the generated prompt to the TUI along with an action card prompting the developer to paste the completion from an external frontier LLM, or checks the environment for a configured fallback API key (e.g., `ADLC_API_KEY`) to run the query directly.
4. **Feed Completions via Stdin (For `--prompt-session` Mode):** Under persistent `--prompt-session` mode, the bridge writes the completions back to the child process's stdin as structured JSON responses:
   `{"type": "prompt_response", "id": "session-uuid", "sample": 0, "round": 1, "response": "..."}`
   The bridge loops stream-drain and stdin-feed operations until the process exits. (For standard `--prompt-only` fallback, the process has already exited, and the bridge simply stashes the completion to be supplied in Phase 2).
5. **Wait for Exit:** The bridge waits for the process to exit. If it exits with code `0`, prompt extraction completes successfully. If it exits with a non-zero code (exit 1 or 2), prompt extraction has failed; the bridge aborts the session, blocks execution, and logs the exit code and stderr error output. To prevent indefinite hangs during the cascade loop, the bridge enforces a strict keepalive timeout of 90 seconds on model client requests; if the model client fails to return a completion within this window, the bridge terminates the child process (`Bun.spawn.kill`), logs a timeout error to the TUI, and exits with code 1 (failing closed).

#### Phase 2: Verdict Resolution & Recording
1. **Tool-Based Verdict Extraction (Single-Shot Fallback):** For standard single-shot gates, the bridge does **not** perform independent custom parsing of the model's completion response in JS. To preserve deterministic validation contracts ([ADLC Principle 1]), the bridge spawns the tool a second time in verification mode by passing the `--verify-completions` flag (a Proposed `@adlc` CLI Specification Extension) and piping the model completions collected in Phase 1 directly to `stdin` as a newline-delimited stream. If the underlying `@adlc` tool does not support the `--verify-completions` flag, the bridge falls back to writing the collected completions to a temporary JSON file and executes the validation tool targeting that file (or pipes them as a standard stdin stream if the tool supports raw stream ingestion), avoiding any bridge-side response parsing. For multi-sample or multi-round tools running under the `--prompt-only` fallback, the bridge feeds all $N$ completions collected in Phase 1 to the tool, allowing the tool itself to execute consensus aggregation, evaluate the mathematical consensus threshold $k$-out-of-$N$, and compute the final verdict deterministically.
2. **Record Verdict & Calibration to Manifest:** The bridge records this verdict as a keyless gate audit event and appends the evidence/metadata directly to `.adlc/manifest.jsonl` using the `adlc gate-manifest record` protocol. The bridge explicitly extracts the current stack calibration settings ($N$ and $k$) and the review-calibration recall score ([ADLC-C8]) from `.adlc/config.json` and attaches them to the manifest record, preserving the C8↔C11 provenance link ([ADLC-C11]).
3. **Enforce Gate Outcome:** If the tool exits with `exit 2` (Fail) or `exit 1` (Error), the bridge sets the execution outcome to blocked (failing closed for enforcing gates like P1, P3, P4, P5, and logging warnings for advisory gates) and aborts the execution.
4. **Proposed `--prompt-session` Cascade Execution:** For multi-sample or multi-round tools executing under the proposed `--prompt-session` protocol, the tool stays alive during Phase 1 and consumes responses incrementally via `stdin`. In Phase 2, the tool writes its final aggregated JSON verdict directly to its output, which the bridge captures, records (with cached calibration score attached) to `.adlc/manifest.jsonl`, and enforces. If the process exits with `exit 2` (Fail) or `exit 1` (Error), it fails closed.

## 5. Creator/Critic Model Separation Target & Frontier-Free Scaling

To prevent sycophancy ([ADLC-Flaw-F2]) and same-model self-review (Principle 3 / E4), the plugin defines a target architecture to isolate the active builder model from the gate models:

> [!IMPORTANT]
> The model routing, keyless automatic prompts, and separation features described in this section are target architectural designs that depend entirely on the proposed OpenCode SDK extensions (`agent.resolve` resolver hooks, dynamic model configuration, isolated programmatic model client prompting `context.client.prompt`, and the prompt UI API for human bypass confirmations). In environments where these extensions are not yet implemented in the harness, the plugin defaults to running the builder and gate tools on the active session model, relying strictly on **fresh-context isolation (Principle 3 / fresh context is the active ingredient)** as the primary security barrier to prevent sycophancy, and falls back to manual API key setup (`ADLC_API_KEY`) or user-prompt copy-paste flows.

- **Model Routing Configuration & Trivial Ticket Cost Routing:** The plugin reads a configuration file `.adlc/config.json` (autogenerated on `/adlc-init` if missing) defining model routing tiers (frontier, mid, cheap). Conforming to ADLC P0, **Trivial ticket builder agents run strictly on cheap/low-tier models**. Even if OpenCode's active session defaults to a frontier model, the plugin enforces that Trivial ticket tasks, mechanical tools, and single-pass prosecution are routed to cheap models to optimize execution costs (acting as an authorized cost-routing exception to the D4 mid-tier default). Bounded/Substantial tickets use mid-tier models, and Architectural tickets utilize frontier models. Regardless of whether the plugin itself is installed globally (via `~/.config/opencode/opencode.json`) or copied project-locally, commands like `/adlc-init` always configure the project-local `.adlc/` workspace context and scaffold local `.opencode/` directories at the repository root where execution occurs.
- **Dynamic Model Routing for Builders (Proposed SDK Extension Specification):** During Phase 2, the `/adlc-decompose` slash command executes `adlc model-router` (or delegates to the decomposition skill), which writes the recommended model selection to a ticket-specific routing file (`.adlc/routing/<ticket-id>.json`). When OpenCode spawns the builder agent, the plugin intercepts the agent creation via the proposed `agent.resolve` resolver hook interface and dynamically sets the agent's model configuration (`model` property) to match the recommended tier.
- **Frontier/Mid-Tier Isolation for Gates (Proposed SDK Extension Specification):** Consistent with ADLC D4, the Phase 5 (Prosecute) subagents and verification loops run on the configured mid-tier/calibrated models by default to scale keylessly without requiring frontier resources. A frontier model (`config.frontierModel`) is requested via `context.client.prompt` for Phase 1 (Interrogate) spec shaping/validation, Phase 2 (Decompose) ticket partitioning (to ensure contract stability, ADLC D4), or for specific Phase 5 categories when calibration benchmarks (`review-calibration`) detect a category-specific blind spot in the mid-tier model. Note that Phase 3 `hollow-test` is a fully mechanical gate that does not execute model prompts.
- **Frontier-Free Scaling via Dynamic Calibration:**
  - If the active model name does not match standard cloud provider routing schemas (e.g. running a local model via Ollama), the plugin runs a startup benchmark using `adlc review-calibration`. To prevent live working-tree mutation and protect the user's active tree from corruption, this tool operates entirely within a cloned git repository in a temporary directory (or a temporary memory-backed git ref), planting synthetic bugs on a historical git diff (retrieved from the repository history) and verifying if they are caught by the model stack, completely bypassing the active working tree.
  - **Fresh Repository Handling:** If the repository has no merge or commit history (e.g., a newly initialized repository), the calibration benchmark is skipped. The plugin falls back to a safe default consensus setting (e.g., $N = 3$ passes under majority consensus) and outputs a warning advising that calibration cannot run until the initial commits are recorded.
  - **Calibration Trigger & Caching:** To avoid the high computational cost of running review-calibration, the plugin caches the calibrated stack size $N$ and consensus $k$ settings in `.adlc/config.json`. The calibration benchmark is completely decoupled from the session-start preflight hook (which strictly verifies environment determinism, [ADLC Appendix F / D2]). Instead, `adlc review-calibration` runs dynamically at the start of Phase 5 (Prosecution) if the model configuration changes or if the cache is missing, completely bypassing the active working tree by executing within the isolated git clone. The configuration change is detected by computing a SHA-256 hash of the active model parameters (provider, identifier, temperature, and system prompt templates) and comparing it against a stored hash in `.adlc/config.json`. If the hash deviates or the cache is missing, calibration is marked stale and re-triggered synchronously. To prevent race conditions and ensure that prosecution does not proceed with stale calibration settings (breaking the C8↔C11 provenance chain), the execution is synchronous and blocks the prosecution fan-out until the calibration benchmark completes and updates `.adlc/config.json` with the verified settings. Because calibration requires running intensive multi-sample benchmarks directly against the target LLM provider, it cannot utilize the keyless cascade stdio bridge. It must be executed as a direct background CLI subprocess utilizing direct provider environment keys (e.g. `ANTHROPIC_API_KEY`, `GEMINI_API_KEY`). If these provider API keys are missing, the tool cannot execute online calibration and falls back to safe default settings (N = 3 passes, majority k = 2 consensus) and writes the "UNCALIBRATED" sentinel to `.adlc/config.json`, warning the user instead of crashing. A crash only occurs if the developer explicitly routes prosecution to a remote frontier model but fails to supply its API key. For verdicts recorded in the manifest before any calibration has successfully executed, the manifest C8 calibration fields are populated with explicit sentinel values (e.g., `-1` or `"UNCALIBRATED"`), which `gate-manifest verify` recognizes as a valid uncalibrated baseline. Specifically, the C8 calibration metrics are formally defined as two separate fields in the manifest schema to prevent semantic ambiguity:
    - `calibrationRecall`: A float in the range [0.0, 1.0] representing empirical stack recall, or the string `"UNCALIBRATED"` as a valid calibration state.
    - `calibrationStackSize`: A positive integer representing the stack size $N$.
    Any other types or formats are rejected during `gate-manifest verify` audit, ensuring unambiguous C8↔C11 provenance tracking.
  - Conforming to ADLC Appendix E2 (Search replaces insight) and C8, the plugin does not analytically extrapolate the stack size $N$ parallel passes using an independence assumption (since same-model high-temperature resampling produces correlated failures, meaning a model that misses a vulnerability class will consistently miss it on all draws). Instead, the plugin **empirically measures the actual fanned-out $N$-pass stack recall score**.
  - The plugin runs `adlc review-calibration` on the $N$-pass stack (starting at $N = 3$) and increments the stack size $N$ until the empirically measured stack recall meets or exceeds the required threshold (e.g., $95\%$).
  - **Consensus Threshold & Specificity Blowup:** A strict unanimous-to-pass rule over $N$ passes drives false blocks (false positives) up as $1 - (1 - f)^N$ under a theoretical assumption of independent draws, where $f$ is the single-pass false-positive rate. However, since same-model resamples are highly correlated, this formula serves as an illustrative worst-case upper bound rather than a precise analytical model. To mitigate specificity blowup in practice, the calibration benchmark evaluates both recall and specificity empirically, and the plugin sets the consensus threshold to require a majority (e.g., $k$-out-of-$N$ consensus where $k = \lfloor N/2 \rfloor + 1$) rather than unanimity, ensuring both high recall and high specificity. The system warns the user that validation depth has been calibrated to $N$ passes under $k$-out-of-$N$ consensus, preserving the zero-key thesis (with key fallbacks documented strictly for missing programmatic APIs).
  - **Consensus vs. Verifier Composition:** The $k$-out-of-$N$ majority consensus governs the *calibration threshold* for local-model scaling at setup time (evaluating if the model is statistically capable), whereas the `prosecutor-verifier` agent handles *individual finding validation* in the active prosecution loop. To ensure the manifest's calibration score accurately describes the active reviewer, the `adlc review-calibration` CLI performs calibration by receiving the exact prosecutor agent templates from the plugin (rather than hardcoding paths to OpenCode's directory layout, avoiding harness-specific assumptions in the shared core) and executing them under the benchmark harness, matching the exact subagent reviewer stack that issues the prosecution verdicts and preserving the C8↔C11 provenance link.
  - **Loop Convergence Budget:** The maximum number of loop rounds for prosecution convergence is denoted by $R_{\text{max}}$ (read dynamically from `maxRounds` in `.adlc/config.json`, defaulting to 5 rounds) to differentiate it from the parallel stack size $N$ (number of parallel sampling passes). If $R_{\text{max}}$ loop rounds are reached without drying out, prosecution fails closed (exits with code 2, failing the gate), blocking integration and preventing any merge until the developer manually resolves the findings or revises the ticket spec/contract.
  - **Orchestration of the P5→P4 Fix Loop:** If the prosecution subagents find verified findings, the `/adlc-prosecute` command continues execution of the P4 builder agent in its existing context (retaining the builder conversation history/context so the builder has access to its prior attempts and build context), appending the structured findings checklist to prompt the builder to repair the code. This conforms to ADLC Appendix F (where fixers *continue* the build context, and only reviewers are fresh). If the harness doesn't support dynamic context continuation, `/adlc-prosecute` exits with code 2, prints the findings as a structured markdown checklist in the TUI, and suspends the session, requiring the developer to manually resume the builder agent to address the checklist before re-running prosecution.
- **Behavior-Diff Project Configuration:** Under the `behaviorDiff` block in `.adlc/config.json`, the project owner configures capture commands and target outputs to record observable system behavior. The schema of this configuration includes:
  - `captureCommand`: The CLI command (e.g. `npm run test:behavior` or a custom script) to run and capture stdout/files.
  - `fixturesDir`: Directory containing fixture input/output payloads to be monitored/diffed.
  - `modalities`: A list of active capture modalities (e.g., `["cli", "api", "events", "browser"]`).
  If this configuration is absent, `behavior-diff compare` defaults to executing the standard test suite configured in `testCommand` to verify behavior remains green, logs an advisory warning, and prompts the developer to define behavior capture commands for complete P6 validation.

## 6. Completeness & Implementation Limitations

All 8 phases of the ADLC are represented in this plan (with Phase 4 (Build) realized natively by OpenCode's active builder agent and gated dynamically in-session by the `rails-guard` and `flail-detector` hooks), but three specific boundaries are partially implemented or restricted by design.

### 6.1. Phase 6 (Integrate) is Inherently Manual (Partial)
* **Status:** ⚠️ Partial Implementation
* **Reason:** Human Gate 2 ("Is this what I meant, running?") is the human's final control gate. While the plugin automates gathering evidence (`gate-manifest record`, `behavior-diff compare` dynamically executing against the `triageCommit` git baseline to establish the behavioral diff baseline, and `review-calibration` scores), it **cannot automate behavioral acceptance**. The plugin surfaces an execution summary card in the TUI, and the developer runs `/adlc-accept --ticket <id>` to record human acceptance.

### 6.2. In-Session Bash Gating is Gated at CI/CD Commit Time (Partial)
* **Status:** ⚠️ Dropped for In-Session Hooks, Gated at CI/CD Commit Time
* **Reason:** Aligning with the command reconciliation decision history, parsing Bash commands in-session is a source of false security. Shell execution is Turing-complete (brace expansions, subshells, aliases, option overrides like `rm -rf`). Gating Bash in-session leads to bypasses.
* **Mitigation:**
  - **CI/CD Rails-Guard Union Strategy:** Any bash-based modifications to frozen rails are caught at commit time using an unbypassable **CI/CD rails-guard check** (`docs/ci/rails-guard.yml` or script `scripts/rails-guard-ci.mjs`) which diffs the branch against the trunk base. Crucially, to prevent deletion bypasses and protect new rails introduced in the same PR, the CI check evaluates the **union of rail declarations** from both the trunk base branch (e.g., `git show origin/main:.adlc/tickets.json` or target base ref) and the current head ref (PR branch). Any rail path present in the base branch `tickets.json` is strictly locked and cannot be deleted or modified by the PR commits (deletion-proof), while any new rail paths declared in the PR branch's `tickets.json` are also enforced as frozen for all implementation changes within that PR (new-rail-aware). To prevent builder agents from using bash in-session to bypass hook checks by modifying the plugin source code or tampering with configurations/manifests, the CI/CD rails-guard check strictly protects the `.opencode/` and `.adlc/` directories: any modifications to `.opencode/plugin/` (including the pre-bundled `dist/index.js` or scripts), `.opencode/opencode.json`, `.adlc/manifest.jsonl`, or `.adlc/config.json` inside the PR commits are blocked. Furthermore, `.adlc/tickets.json` is treated as a frozen rail; the workflow reads the baseline triage commit hash stored as `triageCommit` in `.adlc/tickets.json` (failing closed and rejecting the PR if the baseline commit hash is missing — unless the base branch contains no `.adlc/tickets.json` file at all, which represents the repository's first-ever ticket PR and is exempt from base-comparison checks — prompting the developer to run a ticket repair command if it is a non-first PR), and uses that commit ref as the comparison base ref to ensure that no existing rail definitions for the active ticket were deleted or modified. Any PR that deletes or alters active rail paths in `tickets.json` relative to their initial triage state is strictly rejected.
  - **Local-Only / Non-CI Simplify Fallback:** Since repositories without CI/CD workflows cannot execute the post-merge Simplify pass automatically, the plugin expands the `/adlc-distill` command: running `/adlc-distill --simplify` executes the Simplify pass locally on-demand once all tests are green, protecting the repository against code bloat. This local pre-merge execution is a compromise deviation from the strict post-merge ADLC P7 specification (which is flagged to the developer as an advisory warning on invocation, noting the risk if the ticket's spec is revised prior to merge). To preserve the ADLC P7 invariant (refactoring under green, still-frozen rails), the `/adlc-distill --simplify` execution flow refactors the implementation files under the protection of the target completed ticket's rail paths, which remain strictly frozen and immutable in `.adlc/config.json`. The rails-guard hook enforces absolute edit protection on these rail files themselves during the distillation run, immediately aborting the simplification if any mutations to the tests are attempted, while permitting the builder agent to edit the implementation files to achieve cleanup under the green test suite.
  - **Gate-Fuzzing Sandbox Safety Isolation:** Since `adlc gate-fuzzing` executes untrusted adversary-generated setup and witness code requiring an OS-level sandbox (`bwrap`/`sandbox-exec`), it is **strictly blocked from running on the developer host** during `/adlc-maintain` execution. The plugin TUI command only runs safe model/skill decay checks. `gate-fuzzing` is strictly restricted to execution in CI/CD pipelines (e.g. `ci/adlc-maintenance.yml`) or disposable VM containers where sandboxing is guaranteed. To ensure this security boundary, the `adlc gate-fuzzing` CLI tool itself enforces sandbox verification at execution time, checking for active containerization or OS-level sandbox namespaces, and refusing to run (exiting code 1) if executed directly on a bare developer host. Note that container sandboxing checks are best-effort heuristics (bypassable via host filesystem mounts or custom user configurations), necessitating CI trunk protection backstops.
* **Local-only / GitLab / Pre-commit Hook Hardening:** For repositories without GitHub workflows, a pre-commit git hook (`.git/hooks/pre-commit`) is configured during `/adlc-init`. To prevent clobbering any pre-existing user pre-commit hook or placing the verification after early exits (such as `exit 0`), `/adlc-init` parses the shebang and structure of the existing file. If the shebang specifies a non-shell interpreter (e.g. `#!/usr/bin/env python3` or similar), it does **not** inject shell commands and instead logs a warning or prompts the user for manual configuration. For shell-based pre-commit hooks, it identifies any early exit statements, and injects the `adlc rails-guard` execution at the top of the file (or wraps the existing hooks to ensure `adlc rails-guard` runs first and blocks execution on failure), rather than blindly appending to the end where it could be unreachable. It prompts the user for manual confirmation if it cannot parse the script. To prevent self-modification bypasses, the pre-commit hook executes `adlc rails-guard` reading the union of rail declarations from `HEAD` (`git show HEAD:.adlc/tickets.json`) and the staged index, ensuring new rails are locked locally. We explicitly note that local git hooks are best-effort (as the agent can delete `.git/hooks/pre-commit` via a bash terminal execution) and must be backstopped by branch protection in CI/CD. Local-only developers must manually inspect hook logs and run direct path diff comparisons against the base branch to verify rail compliance before merging.

### 6.3. Background Scheduling (Partial)
* **Status:** ⚠️ Partial Implementation
* **Reason:** Phase 7 (Distill) mines repeated findings and PR rejections into reusable skills and lints. If the developer's OpenCode client runs on a transient host without persistent background daemons, cron-based maintenance or distillation runs are unavailable.
* **Mitigation:** The plugin exposes explicit TUI commands (`/adlc-distill`, `/adlc-maintain`) to run these processes on-demand at the end of a session, supplementing CI-based cron pipelines.

### 6.4. Keyless Prompting & Creator/Critic SDK Dependencies (Partial)
* **Status:** ⚠️ Partial Implementation / Dependent on SDK Extensions
* **Reason:** The keyless prompt extraction bridge, isolated sub-contexts, flail-detector builder restart, and creator/critic model separation rely on proposed OpenCode SDK extension interfaces (`context.client.prompt({ isolated, model })` and `agent.resolve` hooks). Until these APIs are implemented in the harness, the plugin uses fresh-context isolation on the active session model, and requires manual `ADLC_API_KEY` configuration or manual copy-pasting for gate queries.

### 6.5. Local-Only / Non-CI Security Degradation Summary
* **Status:** ⚠️ Advisory
* **Reason:** In workspaces configured without CI/CD branch protection pipelines, the safety contract decays due to the composition of the following six limitations:
  1. **In-Session Bash Exclusion:** Bash shell commands are not gated in-session, allowing edits to rails via standard terminal tools.
  2. **Harness Fail-Open Exceptions:** If the OpenCode harness lacks native `onFailure: deny` support, plug-in loading errors or crashes fail open.
  3. **Agent-Deletable Hooks:** Local pre-commit git hooks are best-effort and can be bypassed or deleted by the agent.
  4. **No Remote Backstop:** Repositories without CI have no secondary trunk-base diff validation to catch bypassed edits.
  5. **Temporal Isolation Bypass:** If the proposed TUI suspension API is absent, builder agents cannot be suspended during `/adlc-rail-write` execution, allowing concurrent builder agents to write to pending rails during this temporal window.
  6. **Evidence Tampering via Bash:** Local evidence files in `.adlc/` are writable via the terminal, enabling agents to falsify test/build logs locally.

  Under this composite, there is **zero complete or guaranteed mechanical rail enforcement** during development. Local-only developers must manually audit git logs for edits to `.adlc/` and run direct path comparisons against the base branch to verify rail compliance before merging.

### 6.6. Shared Hook-Core Refactor Prerequisite
* **Status:** ⚠️ Technical Debt (Resolved Blocker)
* **Reason:** The shared-hook-core factorization `@adlc/core` is already published (v1.0.2) and active in the registry. The remaining task is resolving hook duplication across the Claude Code hook (`hooks/adlc-hook.mjs`) and the Codex hook (`plugins/adlc-codex/hooks/adlc-rails-guard.mjs`), which today run separate implementations of the same rails logic.
* **Mitigation:** The plugin's Phase D implementation is not blocked. We will consume the shipped `@adlc/core` package directly. Refactoring the duplicate legacy code in both the Claude hook and the Codex hook to use the shared core is treated as standard technical debt cleanup to prevent drifting copies of security-critical code.

### 6.7. Lack of Parallel Worktree Orchestration
* **Status:** ⚠️ Omission by Design
* **Reason:** The ADLC parallel orchestration specifications (P4/D0–D2) rely on multi-writer workspace worktrees and automated branch merges to resolve conflicts. OpenCode's client integration targets a single-session interactive terminal.
* **Mitigation:** All ticket execution in OpenCode runs sequentially in a single workspace. Parallel execution is delegated to external orchestrators (like `agb` fleets running on independent runner instances) rather than running concurrently within a single OpenCode environment.

### 6.8. Proposed Runner Extension Fallbacks
* **Status:** ⚠️ Advisory
* **Reason:** If the proposed `@adlc/runner` extensions (`adlc-runner run p0` and `adlc-runner run p4`) are absent or fail:
  - **P0 Triage Fallback:** If `adlc-runner run p0` is unavailable, the plugin falls back to a local mechanical helper script (`node .opencode/commands/triage-check.js`) that directly parses `.adlc/tickets.json` to verify ticket presence and matches the active diff size against the triage limit. Upon success, the helper script automatically records a fallback P0 triage record (and writes any required `auto_passed`/`skipped` records for P1 and P2 if the active ticket is triaged as Bounded or Trivial) to `.adlc/manifest.jsonl` using `adlc gate-manifest record --phase p0 --status pass`, ensuring subsequent verification checks do not fail due to missing evidence.
  - **P4 Build Gate Fallback:** If `adlc-runner run p4` is unavailable, the prosecution command `/adlc-prosecute` falls back to direct execution of the local test suite and linter commands (configured in `.adlc/config.json`), checking their exit codes directly to assert build-gate validation. Upon success, the fallback execution automatically records a fallback P4 build record to `.adlc/manifest.jsonl` (using `adlc gate-manifest record --phase p4 --status pass`), ensuring subsequent verification checks do not fail due to missing evidence.

## 7. Phased Delivery Plan

We propose a six-phase delivery plan, mirroring the successful Claude Code implementation. Each phase will be subjected to an adversarial counter-review loop to check for bypasses:

* **Phase A — Dependency & Commands:** Configure package mappings for `adlc` and `adlc-runner` (pending merge). Implement `/adlc-init` (which scaffolds `.adlc/config.json` and copies/symlinks slash command/agent/skill directories into `.opencode/` from the package source), `/adlc-ticket` (P0 triage interview writer), `/adlc-approve-spec` (P1 G1), and gated slash commands for `/adlc-spec` (P1) and `/adlc-decompose` (P2) to ensure the lifecycle has an entry point.
* **Phase B — The Keyless Bridge:** Write the plugin SDK hook that implements the two-phase stdio JSON cascade protocol to extract prompts via `--prompt-session` (falling back to `--prompt-only` with stdin replay in Phase 2) and route prompts keylessly to isolated model sub-contexts, handling concurrent multi-round and multi-sample flows.
* **Phase C — In-Process Advisory Hooks:** Implement `session.created` (checks runtime/git readiness and environment determinism) and `session.ended` (runs `gate-manifest verify`).
* **Phase D — Enforcing Rail-Guard Hook:** Deploy the `tool.execute.before` hook thin adapter (`rails-checker.ts` importing validation logic from `@adlc/core`) gating structured file edits with parent/segment canonicalization, canonical plugin root path protection, suppression-marker scans (Goodhart defense), and fail-closed return contracts.
* **Phase E — Prosecutor Subagents, G4 Build Gate & Distill:** Configure the G4 build gate verification command (/adlc-verify-build) and runner assertion (adlc-runner run p4) running build/lint/test suite prior to prosecution, the fanned-out prosecution subagents (5 lenses), the verifier/reproducer agent, and loop orchestration. Wire `/adlc-distill` for lesson mining and optional local `--simplify` pass.
* **Phase F — CI Templates & Adoption Guide:** Deliver `.github/workflows` configurations (hardened to read the union of `.adlc/tickets.json` from the trunk base ref and head branch), pre-commit hook scripts (hardened via union of HEAD and staged index), and publish `docs/opencode.md` for users.
