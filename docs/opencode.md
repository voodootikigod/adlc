# Adopt the ADLC in OpenCode

> **Status: proposal / implementation target.**
> This PR documents the intended OpenCode integration contract. It does not ship
> the `@adlc/opencode-plugin` npm package or a `plugins/adlc-opencode` source tree
> yet, so the plugin installation commands below are target commands for the
> implementation PR that adds those artifacts.

The `@adlc/*` toolkit is a set of gate-shaped CLIs. This plugin makes the whole **Agentic Development Lifecycle** usable natively from inside OpenCode: the gates fire at the right lifecycle moments — some automatically — and the model reaches for the right gate without you memorizing 19 separate tools.

Following **Option D (separate, concern-focused bins)** from the Command Reconciliation ADR:
- `adlc` is the dispatcher (`@adlc/cli`) for executing individual gate tools.
- `adlc-runner` is the runner (`@adlc/runner`) for phase-evidence assertions.

> Design and rationale: [`opencode-integration-plan.md`](./opencode-integration-plan.md).
> The full thesis: [`../ADLC.md`](../ADLC.md).

## Install

### 1. Install the ADLC CLI Tools
The plugin needs the ADLC binaries globally resolvable in your PATH. Once published, the preferred installation method is via the unified convenience meta-package:

```sh
npm install -g @adlc/all
```

This package automatically installs the `adlc` dispatcher, `adlc-runner`, and all 19 gate CLIs.

> [!IMPORTANT]
> Because `@adlc/runner` is a pending-merge dependency from the sibling Codex branch, the convenience bundle `@adlc/all` is not yet published to the public registry. You must install the tools locally:
> 1. Clone the Codex branch of the `adlc` repository.
> 2. Build the binaries locally following the guide in the root [README.md](../README.md).
> 3. Link them to your global path (including the dispatcher, runner, and the 19 independent gate packages, ensuring no first-use ENOENT errors occur):
>    ```sh
>    # From the root of your local cloned repository directory:
>    npm link packages/*
>    # Verify that the dispatcher and runner are globally accessible on your PATH:
>    adlc --version
>    adlc-runner --version
>    ```
> Do NOT mock or stub `adlc-runner` in your PATH; doing so silently disables all phase assertions, leaving the workspace completely unprotected.

The plugin initialization check will verify that both `adlc` and `adlc-runner` bins are available on PATH and output a clear installation error if missing.

### 2. Install the OpenCode Plugin
> [!NOTE]
> The plugin manager command (`opencode plugin add`) and configuration file format (`opencode.json` plugin registry mapping) described below are target specifications that must be verified against the official OpenCode SDK specification during implementation. Do not treat them as runnable install instructions until the implementation adds the package or source tree.

Once implemented and published, the expected package install path is:

```sh
opencode plugin add @adlc/opencode-plugin
```

For local development after the implementation lands, copy or symlink the expected plugin source directory (`plugins/adlc-opencode`) to your project-local `.opencode/` directory:

```sh
mkdir -p .opencode/plugin/
# Copy or symlink the integration code:
ln -s /path/to/adlc/plugins/adlc-opencode/ .opencode/plugin/adlc-opencode
```

Alternatively, register the implemented plugin globally in your `~/.config/opencode/opencode.json`:

```json
{
  "plugin": [
    "/path/to/adlc/plugins/adlc-opencode"
  ]
}
```

### 3. Initialize your Repository
Bootstrap the ADLC workspace context in your repository by typing the following slash command directly into the OpenCode TUI chat interface:

```text
/adlc-init
```

This performs the following bootstrap actions:
1. **Directory Structuring:** Creates the `.adlc/` workspace configuration directory, templates the `.adlc/config.json` and `.adlc/tickets.json` structures, writes the required `.adlc/config.json.securityMode` sentinel (`"signed"` or `"unsigned-fallback"`), and appends runtime logs and temp directories idempotently to your `.gitignore`.
2. **Local Bindings:** Scaffolds the local `.opencode/` subdirectories (`command/`, `agent/`, `skill/`) and configuration `opencode.json` (autogenerated if missing).
> **Current limitation:** Until `@adlc/runner` is merged and published and `adlc-runner --version` resolves, generated keys are not used for cryptographic signing. `/adlc-init` must set `.adlc/config.json.securityMode` to `"unsigned-fallback"` in that state, manifest records written through fallback paths are unsigned entries (`"unsigned_fallback": true`), and branch-protection or signed-evidence-required checks must not be configured as enforcing. The cryptographic evidence chain described below is the design target, not current behavior.
3. **Developer and Admin Key Generation (Security-Critical Proposed `@adlc/runner` Extensions):** Runs the proposed `adlc-runner init-developer-keys` extension to generate a persistent developer private key pair (specifically Ed25519 as the sole canonical developer key algorithm) at `~/.config/opencode/keys/developer.key`. This key is used to sign triage upgrades and manifest evidence. On first bootstrap, `/adlc-init` also runs the proposed `adlc-runner init-admin-keys` ceremony when `.adlc/admin.pub` is absent: it writes the admin override private key to `~/.config/opencode/keys/admin.override` with `0400` permissions, writes no admin private material into the repository, and writes the admin public key fingerprint to `.adlc/admin.pub` so it can be committed in the initial ADLC bootstrap commit. If either proposed runner extension is missing or fails, `/adlc-init` falls back to Node's native `crypto` module, enforcing the same key locations, `0600`/`0400` private-key permissions, `0644` public-fingerprint permissions, and fingerprint registration. The runner checks permissions on startup, refusing to execute signed commands and failing closed if private key permissions are wider than owner-only (`0600`/`0400`). Losing the developer key prevents cross-session manifest signature verification, requiring a cryptographically signed admin manual override. To generate the admin override, the developer signs a JWT challenge token using the admin private key stored out-of-band in `~/.config/opencode/keys/admin.override` (isolated from the repository). The signature is verified against the frozen admin public key fingerprint committed at `.adlc/admin.pub` within the repository's base history, preventing local modification bypasses or `.git/config` tampering. In a single-developer setup, this key acts as a secure, offline recovery mechanism, verified against the frozen public key fingerprint, with compromised keys checked against a local/global revocation list `~/.config/opencode/revocation_blocklist`. To protect against compromised developer keys, a `revokedKeys` fingerprint revocation blocklist can be declared in `.adlc/config.json`, which `adlc-runner` checks to reject signatures from revoked keys. Note that if the developer private key file is deleted or missing after initialization but other configs exist, executing `/adlc-init` will NOT overwrite or regenerate the key (to prevent invalidating the manifest chain signatures). Instead, the plugin fails closed, and the developer must recover/rotate the key by running `adlc-runner rotate-developer-keys --recover --admin-key <path>` (authorized via the admin override key).
4. **CI/CD Workflow Scaffolding:** Copies template workflows from `docs/ci/rails-guard.yml` and `docs/ci/adlc-maintenance.yml` to `.github/workflows/`. `/adlc-init` first verifies that both source templates exist in the installed package and fails closed with a clear installation error if either template is missing, preventing silent loss of the CI rail backstop. Required branch-protection checks are scaffolded only after a precondition checklist confirms `adlc-runner --version` resolves, signed manifest commands are available, and `.adlc/config.json.securityMode` is `"signed"`; otherwise the workflows may be installed only as advisory checks. The current rails-guard template protects rails already present on the base branch and, after bootstrap, protects immutable ADLC trust roots, CODEOWNERS self-protection, signed-runner configuration, existing ticket contracts, and append-only manifest evidence. It does not protect rails introduced for the first time in the same PR, which remain a future new-rail-aware union-gate requirement. Until that gate ships, `/adlc-init` must require `--acknowledge-new-rail-bypass` in headless mode or an interactive TUI confirmation before scaffolding `rails-guard.yml` as a required check. If destination workflow files already exist, `/adlc-init` will prompt you before overwriting, allowing manual merging.
5. **Hook Installation:** Installs a local pre-commit hook (`.git/hooks/pre-commit`) for environments that lack CI runners, parsing existing hooks to ensure it does not clobber them and runs its validation first.

### Headless/CI and Non-Interactive Triage
For headless or non-interactive environments (such as CI/CD pipelines), the plugin supports automated triage mapping via the `ADLC_TRIAGE_CLASS` environment variable. You can set it to `Trivial`, `Bounded`, `Substantial`, or `Architectural` to bypass the TUI triage interview. If unset, it defaults to `Substantial` (which enforces full specification and decomposition gates, which may fail/deadlock in headless environments if spec inputs are missing). **Precedence rule:** To prevent bypass reward-hacking, the `ADLC_TRIAGE_CLASS` environment variable and `--class` flags only apply during initial triage creation when no entry exists in `.adlc/tickets.json` for the active ticket. Once the triage classification is recorded, any subsequent environment override or downgrading attempts are strictly ignored, and the plugin compares active parameters; any deviation triggers a config mismatch warning and fails closed to prevent downgrade attacks.

---

### 4. Configure LLM Provider API Keys & Frontier Models
Most ADLC gates execute keylessly using OpenCode's active session model. However, to ensure safety boundaries:
1. **Frontier Model Requirement for P1 and P2:** Consistent with ADLC.md, Phase 1 (Interrogate) spec shaping and Phase 2 (Decompose) ticket partitioning require a frontier-tier (Opus-class) model (such as an active frontier-class model) to prevent underspecification and guarantee contract desk stability. Mid-tier (Sonnet-class) and cheap (Haiku-class) models are suitable for Phase 5 (Prosecution) or Phase 4 (Build), but do not cost-optimize Phase 1 or Phase 2 by routing them to mid-tier or cheap models.
2. **Calibration API Keys:** Phase 5 prosecution requires dynamic benchmark calibration (`adlc review-calibration`) which must execute directly against the LLM provider. You must export the corresponding API key (e.g. `ANTHROPIC_API_KEY`, `GEMINI_API_KEY`) in your environment. Failure to configure these keys will cause prosecution to fall back to safe default settings (N = 3 passes, majority k = 2 consensus) and write the "UNCALIBRATED" sentinel, raising a warning notification instead of crashing (unless the developer has explicitly configured a remote frontier model for prosecution/calibration without its API key, which will cause a startup crash).

(2) *Type Exports*: The plugin parses the active git diff to check if files with type-definition extensions (such as `.d.ts`, `types.ts`, `interface.ts`, or public API export boundaries like `index.ts`, `public-api.ts`, or schema files like `schema.graphql`, `openapi.yaml`) are modified, or if the diff introduces changes to exported symbols or public-facing function signatures (via AST or regex scanning for `export`, `public`, `interface`, `type` keywords). ### The `adlc-runner` CLI Reference

Under the concern-focused separate binary architecture (**Option D**), while the `adlc` dispatcher runs individual gate tools, `adlc-runner` is the security-critical runner (`@adlc/runner`) responsible for managing cryptographic keys, recording manifest entries, and asserting phase evidence.

#### Key Management Subcommands

##### `adlc-runner init-developer-keys`
* **Description:** Generates a persistent developer cryptographic key pair (specifically Ed25519) locally in the developer's user configuration directory.
* **Usage:** Run automatically during `/adlc-init` bootstrapping.
* **Security Controls:**
  - Writes the private key to `~/.config/opencode/keys/developer.key` with secure, restricted owner-only permissions (`0600`/`0400`).
  - Writes the public key fingerprint to `~/.config/opencode/keys/developer.pub` (permissions `0644`) and registers it in `.adlc/config.json`.
  - Performs startup validation to reject the private key and fail closed if POSIX permissions are wider than owner-only, or if the file owner UID does not match the active shell execution context.
  - *Fallback:* If the subcommand is unavailable, the `/adlc-init` bootstrapper falls back to Node's native `crypto` module to generate the keys with identical permissions and paths.

##### `adlc-runner init-admin-keys` (Proposed `@adlc/runner` Extension)
* **Description:** Generates the out-of-band admin override key pair used for key-loss recovery.
* **Usage:** Run automatically during the first `/adlc-init` bootstrap when `.adlc/admin.pub` is absent.
* **Security Controls:**
  - Writes the admin private key to `~/.config/opencode/keys/admin.override` with `0400` permissions.
  - Writes the public fingerprint to `.adlc/admin.pub` for commit in the initial ADLC bootstrap commit.
  - Refuses to overwrite an existing `admin.override` or `.adlc/admin.pub` unless an explicit rotation/recovery ceremony is used.
  - *Fallback:* If the subcommand is unavailable, `/adlc-init` falls back to Node's native `crypto` module with the same file locations, permissions, and fingerprint checks.

##### `adlc-runner rotate-developer-keys [--recover --admin-key <path>]` (Proposed `@adlc/runner` Extension)
* **Description:** Rotates the active developer cryptographic key pair while maintaining manifest chain of trust integrity.
* **Standard Rotation:** Generates a new developer key pair, appends the new public key fingerprint to `.adlc/config.json`, and signs a `key_rotation` transition event in the manifest using the old key.
* **Key Loss Recovery (`--recover --admin-key <path>`):** In the event of post-initialization key loss (where the runner fails closed, blocking edits), this recovery path generates a new developer key pair, authorized via an out-of-band admin override key stored at `~/.config/opencode/keys/admin.override` (isolated from the repository). It verifies the admin key signature against the frozen public key fingerprint committed in the base repository history, updates `.adlc/config.json` with the new public key, and appends a signed rotation event to the manifest (exemption: this recovery command is explicitly permitted to write to the manifest in the fail-closed state since it is authorized by the verified admin key).

#### Phase Assertion & Manifest Subcommands

##### `adlc-runner run <phase>`
* **Description:** Assert that a specific lifecycle phase's requirements and evidence are satisfied and recorded in the manifest.
* **Phases:**
  * **`p0` (Triage):** Asserts that: (1) a valid ticket entry exists in `.adlc/tickets.json`, (2) the `triageCommit` is present in `.adlc/tickets.json` and is either a valid, reachable commit in the repository's git history or matches the sentinel `"EMPTY_REPO_INITIAL"` (which skips the git reachability check for freshly initialized repositories with no history), (3) active diffs do not violate mechanical triage classification limits (e.g. 3 files / 50 lines for Trivial) — to evaluate the active diff without crashing when `triageCommit = "EMPTY_REPO_INITIAL"`, the check must fall back to evaluating untracked/added files in the current working directory using `git status --porcelain` (or diffing against the empty git tree hash `4b825dc642cb6eb9a060e50d70f22db4b4a70777`), and (4) all parent/foundation ticket dependencies are resolved and merged first, blocking ticket activation if parent tickets remain incomplete. For Bounded/Trivial tickets, a successful P0 triage assertion automatically writes the P1 and P2 (and P3 for Trivial) `auto_passed`/`skipped` placeholder records to `.adlc/manifest.jsonl` to establish the manifest sequence base.
  * **`p1` (Spec Interrogate):** Asserts that the spec audit evidence and Human Gate 1 (spec approval) signature are recorded.
  * **`p2` (Decomposition):** Asserts that atomic, typed ticket partitions exist.
  * **`p3` (Rail Freeze):** Asserts and verifies that the rail-freeze evidence (logs and outputs of `hollow-test`, `test-audit`, and compiler typechecks completed during `/adlc-rail-write`) is present and valid in the workspace and manifest, and that the rail files are now frozen. *Fallback:* If the proposed `test-audit` tool is absent, the runner accepts the combination of `hollow-test` logs and the local static analysis script output as a degraded, best-effort fallback approximation rather than an "equivalent proof", acknowledging the loss of LLM-backed detection for mock assertion semantics under this fallback path.
  * **`p4` (Build Gate):** Asserts build gate G4 compliance (compiled outputs, lint cleanliness, passing rail tests, and that frozen rails remain intact with no new unwhitelisted suppression markers).
  * **`p5` (Prosecution):** Asserts that prosecution loops ran dry and verifies the signed G4/P4 build record in the manifest instead of re-running the test suite. For non-Trivial tickets (Bounded, Substantial, Architectural), it also strictly asserts that all rails are green and the `rails-diff-empty` proof exists in the manifest (which are omitted/skipped for Trivial tickets since they contain no rails and skip Phase 3). The p5 check asserts that the prosecution loop (or the single prosecution pass for Trivial tickets) returned zero findings.
  * **`p6` (Integrate):** Not run via `adlc-runner run p6`. Instead, Phase 6 evidence is verified and written via [adlc-runner accept --ticket <id>](#adlc-runner-accept---ticket-id).
  * **`p7` (Distill):** Asserts that Phase 7 distillation evidence (lesson foundry outputs) is present.

##### `adlc-runner record --entry <payload>` (Proposed `@adlc/runner` Extension)
* **Description:** Cryptographically signs and appends a structured evidence or execution log entry to `.adlc/manifest.jsonl`.
* **Details:**
  - Loads `developer.key` (enforcing POSIX permission checks), signs the payload, and appends the entry.
  - Serves as the underlying cryptographic signature engine for the runner's manifest logging operations.
  - Implements a hash-chained structure where each entry contains its own signature, the developer's public key fingerprint, and the SHA-256 hash of the preceding entry.
  - Serializes all writes to `.adlc/manifest.jsonl` using a platform-agnostic file lock on `.adlc/manifest.lock` via `proper-lockfile` to prevent interleaved write concurrency hash corruption.
  - *Absent Runner Fallback:* If the runner package is completely absent under Option D, or if it does not yet implement the proposed `record` subcommand (which exits with a non-zero code or unknown-command error), the plugin catches the failure. In either case, the plugin writes *unsigned fallback entries* (containing an empty signature `""` and an `"unsigned_fallback": true` flag) to `.adlc/manifest.jsonl` using a simple node helper (`node .opencode/commands/unsigned-record.js`) which serializes the file writes via `proper-lockfile`. This warns reviewers/CI but permits advisory local execution, preserving minimal-dependency guarantees. When `gate-manifest verify --complete` (or `--in-progress`) validation is run, the state machine evaluates unsigned fallback entries by verifying if the runner package is indeed completely absent from the environment/repository. If verified to be absent, validation passes with an advisory warning; if the runner package is present, any unsigned fallback entry is treated as a validation failure (fails closed, rejecting the PR/build) to prevent bypass.

##### `adlc-runner upgrade --ticket <id> --key <path>` (Proposed `@adlc/runner` Extension)
* **Description:** Records a signed triage upgrade event in the manifest.
* **Lock Acquisition Order:** To prevent deadlocks when writing upgrade events, the runner strictly adheres to the canonical lock acquisition order: it must acquire `.adlc/tickets.lock` first, and then `.adlc/manifest.lock` second, releasing them in reverse order.
* **Usage:** Invoked automatically by the plugin's `triage-upgrade.js` script when a Trivial/Bounded ticket exceeds its mechanical diff limits or fails post-edit validation (e.g., hollow-test failures).
* **Details:** Signs the upgrade using `developer.key`, updates the triage class to Substantial in `.adlc/tickets.json`, stashes active changes programmatically, and routes the ticket back to Phase 1 spec synthesis. This invalidates prior Trivial records in the manifest, forcing the verification state machine to require full Substantial phase evidence. To prevent concurrent writes and preserve isolation, builder agent execution is strictly suspended during the rail-unfreeze/refreeze transition window (even when using degraded fallback paths).
* **Fallback:** If this subcommand is missing/unsupported (e.g. runner present but subcommand has not shipped yet) or if the runner package is completely absent, the plugin's `triage-upgrade.js` script falls back to executing the upgrade steps programmatically: updating `.adlc/tickets.json` via node `fs`, recording an unsigned `triage_upgrade` event record via `node .opencode/commands/unsigned-record.js` to `.adlc/manifest.jsonl`, performing the programmatic git stash and git restore operations to isolate current changes, halting builder execution, and logging a prominent advisory warning in the TUI/terminal.

##### `adlc-runner accept --ticket <id>` (Proposed `@adlc/runner` Extension)
* **Description:** Finalizes the Phase 6 Human Gate 2 (behavioral acceptance) ceremony and appends the human acceptance signature to the manifest.
* **Details:**
  - Strictly validates that the working tree is clean (having no uncommitted changes in tracked files) and fails closed (rejects execution) if the worktree is dirty. Note that `.adlc/behavior-diff.json` is gitignored (added to `.gitignore` during `/adlc-init`). The clean working tree verification check ignores only gitignored files inside `.adlc/`, namely `.adlc/behavior-diff.json`, `.adlc/*.lock`, and ephemeral `.adlc/*.log` files, preventing behavioral evidence generation from blocking the check. Tracked `.adlc/` files such as `.adlc/tickets.json`, `.adlc/config.json`, and `.adlc/manifest.jsonl` are not exempt; staged or unstaged modifications to them must fail the check.
  - Verifies that valid, signed Phase 5 prosecution evidence exists matching the active commit (asserting that the HEAD commit hash at the time prosecution evidence was recorded is identical to the commit hash being accepted, rejecting any unsigned or mismatched commit hashes).
  - Acquires `.adlc/behavior-diff.lock` and runs the behavior-diff comparison internally before signing. The runner does not trust a pre-existing `.adlc/behavior-diff.json`; any previously present file is treated only as review output and is overwritten atomically after fresh evidence is generated under the lock.
  - Validates the in-memory behavior-diff evidence before writing or signing it. The schema of this evidence record must strictly contain:
    - `activeCommit`: string (HEAD commit hash)
    - `triageCommit`: string (baseline triage commit hash)
    - `timestamp`: string (ISO timestamp)
    - `fixtures`: array of `{ name, baselineHash, currentHash, status }` (At least one entry in the `fixtures` array or a non-empty `captureOutput` string is required to constitute valid behavioral evidence in `behavior-diff.json`. If both `fixtures` is empty and `captureOutput` is empty or null, the runner rejects the schema validation and fails closed.)
    - `captureOutput`: string (stdout log or diff summary)
    - `exitCode`: number (must be `0` for success)
    The runner asserts that `activeCommit` matches the current HEAD commit hash, `triageCommit` matches the ticket's triage commit, and `exitCode` is `0`.
  - Atomically writes the freshly generated evidence to `.adlc/behavior-diff.json`, computes its SHA-256 hash while still holding `.adlc/behavior-diff.lock`, records/embeds this hash directly in the signed manifest entry payload (`phase_acceptance` details), and appends the signed human signature entry to `.adlc/manifest.jsonl` (note that the runner subcommand itself is completely non-interactive and does not issue prompts, allowing it to be invoked programmatically or in headless runs; all interactive prompting is handled by the plugin's slash command `/adlc-accept` or local wrapper scripts).

### The Prosecutor Subagents (Phase 5)

Pre-merge prosecution fanned out across five independent, fresh-context, single-lens subagents to prevent context rot. Consistent with ADLC D4, these subagents run on mid-tier, calibrated models by default, only adding frontier lenses if calibration benchmarks show a category blind spot:
1. **`correctness`**: Off-by-one errors, error swallowing, concurrency, and logic races.
2. **`security`**: Threat modeling, injection vectors, hardcoded secrets, and unsafe input sanitization.
3. **`contract`**: Module boundary validation and interface contract matching. *For Trivial tickets lacking formal decomposition contracts, this lens validates the modified files against the existing type exports and import relationships of the codebase to prevent unauthorized module boundary crossings.*
4. **`diff`**: Compares implementation against the spec to verify all criteria are met. *For Trivial tickets where no specification is synthesised, this lens compares the implementation changes directly against the ticket description and requirements recorded in `.adlc/tickets.json`. If it detects any changes out of scope of the ticket description's intent, it flags them as out-of-scope edits, failing the prosecution pass and requiring the ticket to be upgraded to Substantial to adjust the specification and contract.*
5. **`tests`**: Audits unit tests added by the builder during the build phase (ensuring they assert behavior instead of mocking reality).

Findings are verified by a separate `prosecutor-verifier` reproducer agent. The loop executes until two consecutive passes run completely dry (zero verified findings) to certify coverage, up to the maximum convergence budget $R_{\text{max}}$ read dynamically from the workspace configuration (`.adlc/config.json`). If the budget is exhausted without drying out, the loop terminates and escalates to the developer to revise the specification or implementation partition.

#### Rail Gating Safety & Bypasses
- **Conditional Fail-Closed Guarantee:** If a ticket declares rails and the installed OpenCode SDK supports `onFailure: deny`, operational hook errors fail closed and block editing. If the SDK lacks `onFailure: deny`, preflight blocks enforcing-mode startup unless the repository explicitly sets `allowAdvisoryHooks: true`; in that advisory-only mode, the hook must be presented as non-blocking and the safety guarantee relies on CI/CD backstops.
- **Pending vs. Frozen Rails:** To enforce creator/critic isolation in environments lacking native agent-identity verification APIs (preventing builder agents from writing their own tests), the hook **blocks all rail edits during builder sessions entirely**. Pending rails can only be written during a dedicated `/adlc-rail-write` invocation state. To prevent a fail-open state where a crash or force-quit during `/adlc-rail-write` leaves the temporal token active permanently, the preflight cleanup routine checks and invalidates stale tokens on `session.created` and `session.start` (including reconnect/resume events) by verifying the status of the platform-agnostic file lock (`.adlc/rail-write.lock` via `proper-lockfile`). If the lock is stale (the lock file exists but is no longer actively held by any running process), the hook resolves this stale state atomically to prevent TOCTOU races between concurrent sessions: it first acquires an exclusive file lock on `.adlc/tickets.lock` (using `proper-lockfile` to serialize concurrent preflight hooks), re-verifies the staleness of `.adlc/rail-write.lock`, and only then invalidates and clears the stale session token from `.adlc/tickets.json`, releasing the tickets lock afterward and restoring the strict rail-guard posture. To prevent temporal bypasses, the plugin TUI suspends active builder session execution while `/adlc-rail-write` is running. Once Phase 3 is complete, rails are marked Frozen and blocked for all agents.
- **Isolated Prompting & Stdio Cascading:** Prompts from LLM-backed gates are queried in isolated sub-contexts. Multi-sample, multi-round prompts bubble up keylessly via the structured Two-Phase Stdio JSON Cascade Protocol (`--prompt-session` flag). To support concurrent fanned-out subagents (such as P5 lenses) without collisions, the cascade protocol enforces a unique `correlationId` (UUID) in prompt requests and response envelopes, enabling correct routing across child processes. Under the `--prompt-only` fallback, Phase 2 re-spawns the tool and pipes the gathered completions back to stdin, ensuring all consensus aggregation remains in the deterministic tool. The bridge enforces strict escaping, sanitization, and verification of model text outputs before embedding them inside the JSON stdin response envelopes to prevent prompt-injection attacks. Verdict records explicitly attach the active review-calibration recall score ([ADLC-C8]) to preserve the C8↔C11 provenance chain.
- **Developer Key & Behavior-Diff Configuration:** Cryptographic signatures for upgrades and all manifest evidence records (both human gate signatures and automated execution logs) are signed using a developer key stored at `~/.config/opencode/keys/developer.key` (enforcing POSIX `0600`/`0400` permissions and UID verification). If this developer key is lost or deleted post-initialization, the plugin and runner fail closed (blocking edits/tool execution), requiring backup restoration or rotation via `adlc-runner rotate-developer-keys --recover --admin-key <path>` (authorized via the admin override key, which is explicitly exempt from the fail-closed manifest-write block to prevent recovery deadlocks). Every manifest record is cryptographically signed using the developer's key via `adlc-runner record --entry <payload>` at tool completion to verify provenance and maintain a tamper-evident hash-chained append-only evidence chain where each entry links to the hash of the preceding one. To prevent write concurrency issues and protect hash chain integrity from concurrent writes across hooks, keyless bridges, and upgrade scripts, all write and append operations to `.adlc/manifest.jsonl` (including the execution of `adlc-runner record`) are serialized using a platform-agnostic file lock on `.adlc/manifest.lock` managed via `proper-lockfile`. Key compromise requires a `revokedKeys` fingerprint revocation blocklist in `.adlc/config.json`, which is verified by `adlc-runner` to reject signatures from revoked keys. To prevent unauthorized signing of gates in multi-developer environments, the workspace configuration `.adlc/config.json` defines a `signers` authorization matrix mapping developer public key fingerprints to specific roles (e.g. `builder`, `critic`, `approver`). During `gate-manifest verify --complete` validation, the state machine strictly asserts that the keys signing the G1 (spec approval) and G2 (behavioral acceptance) manifest entries are authorized for the `approver` role, rejecting the validation if any signature is from an unauthorized key. Observable behavior validation in Phase 6 runs `behavior-diff compare` using commands and fixture locations configured under the `behaviorDiff` block in `.adlc/config.json` (such as `captureCommand` and `fixturesDir`). If unspecified, it falls back to the default `testCommand` suite, logging an advisory warning to define explicit behaviors. If both `captureCommand` and `testCommand` are absent from `.adlc/config.json` (or if no test runner is configured), the command **fails closed (blocks integration with exit code 2)**, logging a critical configuration error.
- **Version Compatibility & Preflight Cleanup:** To prevent silent failures during manifest record operations and stale bundled hook logic, the preflight hook (on `session.created` and `session.start`) compares the bundled version of `@adlc/core` against the installed version of `adlc-runner` (via `adlc-runner --version`) and, when registry access exists, the latest available `@adlc/core`. If the bundled core has a major/minor/disallowed patch mismatch, is more than one patch release behind, has missing bundled-version metadata, or is older than `maxBundleAgeDays` (default `7`), the plugin **fails closed (blocks execution)** and displays a critical warning advising that the plugin bundle must be rebuilt to match the active runner API schema and current security boundary. Additionally, to prevent storage leakage from interrupted triage-time test verifications or behavioral checks, the preflight cleanup routine executes `git worktree list --porcelain` to locate the actual directory paths of temporary git worktrees (created in the workspace directory under `.worktrees/adlc-verify-XXX` and `.worktrees/adlc-behavior-XXX` and ignored in `.gitignore`) prefixed with `adlc-verify-` or `adlc-behavior-`. It cleans them up by executing `git worktree prune` and deleting their directory paths. For separate directory clones, it scans the `.worktrees/` folder to locate and delete orphaned clones or directories prefixed with `adlc-verify-` or `adlc-behavior-` that are older than 1 hour or no longer associated with active running processes.
- **Frontier-Free Scaling:** Local models that fail dynamic calibrations are scaled up to N-pass parallel checks (ADLC Exploit E1 sampling diversity) and require majority consensus thresholds ($k$-out-of-$N$), dynamically benchmarked during Phase 5 initialization. If a local model name is unrecognized, it undergoes a lightweight capability probe (syntax/logic check) before dynamic calibration. Calibration (`review-calibration`) runs synchronously (blocking prosecution to prevent race conditions and preserve the C8↔C11 provenance chain) utilizing direct provider environment keys rather than keyless cascade stdio.
- **Absolute Rail Protection (No Bypass):** Conforming to ADLC Principle 5, there is **no bypass mechanism** (such as `ADLC_RAILS_BYPASS=1`) permitted for active rails. If a developer needs to modify rails, they must update the ticket contract or upgrade the ticket class. The hook checker strictly denies all edits to protected paths unconditionally, preventing reward-hacking agents from attempting to bypass enforcement.
- **API Key Alias:** The environment variable `ADLC_API_KEY` is a plugin-level configuration and alias that maps to the active provider key (e.g. `ANTHROPIC_API_KEY`, `GEMINI_API_KEY`), allowing local fallback tools to execute using the correct provider key when the harness programmatic model API is absent or disabled.
- **CI/CD / Local Backstop:** Shell-based rail mutations are blocked at commit-time via GitHub workflows or the local pre-commit hook configured during `/adlc-init`. The workflow template files for GitHub are delivered in `docs/ci/rails-guard.yml` and `docs/ci/adlc-maintenance.yml`; `/adlc-init` verifies both source templates exist before scaffolding them into `.github/workflows/`. Branch-protection required checks must be gated behind `adlc-runner --version` and `.adlc/config.json.securityMode == "signed"` when they depend on signed evidence; signed mode requires an `ADLC_RUNNER_PATH` GitHub Actions secret pointing to an audited runner binary by absolute path and a matching `runnerBinarySha256` field in `.adlc/config.json`. Repositories in `"unsigned-fallback"` mode may run those workflows only as advisory checks. The current `rails-guard.yml` template enforces already-frozen rails from the trusted trunk-base `.adlc/tickets.json` and does not trust PR-head rail globs as `adlc rails-guard` inputs. After bootstrap it also rejects immutable trust-root edits or renames, signed-mode downgrades, signer additions, revocation/security-pattern removals, existing ticket contract changes, manifest rewrites, and non-empty initial manifest evidence created by ordinary PRs. Remaining deferred CI hardening is limited to new-rail-aware union enforcement, `.adlc/admin.pub` replacement blocking, `.opencode/plugin/` and `.opencode/opencode.json` protection, and `adlc gate-manifest verify --complete`; `/adlc-init` must require explicit operator acknowledgement before installing the current base-branch rail template as a required check. Post-bootstrap signer changes require a protected-base admin ceremony countersigned by an existing approver because `.adlc/config.json` is a rail trust root; signer additions are not merged through ordinary PRs. First-time bootstrap PRs must not include pre-populated `.adlc/manifest.jsonl` evidence. The local hook remains best-effort and must be backstopped by branch protection in CI/CD. Local-only developers must manually inspect hook logs and run direct path comparisons against the base branch to verify rail compliance before merging.

#### Local-Only / Non-CI Security Degradation (Warning)

> [!WARNING]
> In workspaces configured without CI/CD branch protection pipelines, the safety contract decays due to the composition of the following eight limitations:
> 1. **In-Session Bash Exclusion:** Bash shell commands are not gated in-session, allowing edits to rails via standard terminal tools.
> 2. **Harness Fail-Open Exceptions:** If the OpenCode harness lacks native `onFailure: deny` support, plug-in loading errors or crashes fail open.
> 3. **Agent-Deletable Hooks:** Local pre-commit git hooks are best-effort and can be bypassed or deleted by the agent.
> 4. **No Remote Backstop:** Repositories without CI have no secondary trunk-base diff validation to catch bypassed edits.
> 5. **Temporal Isolation Bypass:** If the proposed TUI suspension API is absent, builder agents cannot be suspended during `/adlc-rail-write` execution or during triage class upgrade transition windows (Trivial/Bounded → Substantial), allowing concurrent builder agents to write to pending/unfrozen rails during these temporal windows.
> 6. **Evidence Tampering via Bash:** Local evidence files in `.adlc/` are writable via the terminal, enabling agents to falsify unsigned test/build logs locally. Phase 6 behavior evidence is not accepted from a pre-existing file: `adlc-runner accept --ticket <id>` regenerates it under `.adlc/behavior-diff.lock`, atomically writes and hashes the fresh file, and signs only that runner-generated hash.
> 7. **Admin Public Key Replacement Attack:** Without trunk protection, an adversary or compromised agent can perform interactive history rewriting (git rebase) to modify the initial commit, replace the admin public key fingerprint (`.adlc/admin.pub`), forge a matching admin override token, and rebuild the manifest chain with arbitrary signatures. This completely compromises the cryptographic root of trust for key recovery, highlighting why trunk protection on base branches is a mandatory security control.
> 8. **Security Mode Downgrade via Bash:** The repository-local `.adlc/config.json.securityMode` declaration is writable through terminal commands outside in-session hook control. The CI template protects the base config file and rejects signed-to-unsigned downgrades, but local-only workflows must manually audit config diffs before trusting signed evidence.
>
> Under this composite, there is **zero complete or guaranteed mechanical rail enforcement** during development. Local-only developers must manually audit git logs for edits to `.adlc/` and run direct path comparisons against the base branch to verify rail compliance before merging.
