# Adopt the ADLC in OpenCode

The `@adlc/*` toolkit is a set of gate-shaped CLIs. This plugin makes the whole **Agentic Development Lifecycle** usable natively from inside OpenCode: the gates fire at the right lifecycle moments — some automatically — and the model reaches for the right gate without you memorizing 19 separate tools.

Following **Option D (separate, concern-focused bins)** from the Command Reconciliation ADR:
- `adlc` is the dispatcher (`@adlc/cli`) for executing individual gate tools.
- `adlc-runner` is the runner (`@adlc/runner`) for phase-evidence assertions.

> Design and rationale: [`opencode-integration-plan.md`](./opencode-integration-plan.md).
> The full thesis: [`../ADLC.md`](../ADLC.md).

## Install

### 1. Install the ADLC CLI Tools
The plugin needs the ADLC binaries globally resolvable in your PATH. Once published, the preferred installation method is via the unified convenience meta-package:

```sh
npm install -g @adlc/all
```

This package automatically installs the `adlc` dispatcher, `adlc-runner`, and all 19 gate CLIs.

> [!IMPORTANT]
> Because `@adlc/runner` is a pending-merge dependency from the sibling Codex branch, the convenience bundle `@adlc/all` is not yet published to the public registry. You must install the tools locally:
> 1. Clone the Codex branch of the `adlc` repository.
> 2. Build the binaries locally following the guide in the root [README.md](../README.md).
> 3. Link them to your global path (including the dispatcher, runner, and the 19 independent gate packages, ensuring no first-use ENOENT errors occur):
>    ```sh
>    # From the root of your local cloned repository directory:
>    npm link packages/*
>    # Verify that the dispatcher and runner are globally accessible on your PATH:
>    adlc --version
>    adlc-runner --version
>    ```
> Do NOT mock or stub `adlc-runner` in your PATH; doing so silently disables all phase assertions, leaving the workspace completely unprotected.

The plugin initialization check will verify that both `adlc` and `adlc-runner` bins are available on PATH and output a clear installation error if missing.

### 2. Install the OpenCode Plugin
> [!NOTE]
> The plugin manager command (`opencode plugin add`) and configuration file format (`opencode.json` plugin registry mapping) described below are target specifications that must be verified against the official OpenCode SDK specification during implementation.

You can install the plugin either by adding the published npm package `@adlc/opencode-plugin` directly via OpenCode's plugin manager:

```sh
opencode plugin add @adlc/opencode-plugin
```

Alternatively, for local development, copy or symlink the plugin source directory (`plugin/adlc-opencode`) to your project-local `.opencode/` directory:

```sh
mkdir -p .opencode/plugin/
# Copy or symlink the integration code:
ln -s /path/to/adlc/plugin/adlc-opencode/ .opencode/plugin/adlc-opencode
```

Alternatively, register it globally in your `~/.config/opencode/opencode.json`:

```json
{
  "plugin": [
    "/path/to/adlc/plugin/adlc-opencode"
  ]
}
```

### 3. Initialize your Repository
Bootstrap the ADLC workspace context in your repository by typing the following slash command directly into the OpenCode TUI chat interface:

```text
/adlc-init
```

This performs the following bootstrap actions:
1. **Directory Structuring:** Creates the `.adlc/` workspace configuration directory, templates the `.adlc/config.json` and `.adlc/tickets.json` structures, and appends runtime logs and temp directories idempotently to your `.gitignore`.
2. **Local Bindings:** Scaffolds the local `.opencode/` subdirectories (`command/`, `agent/`, `skill/`) and configuration `opencode.json` (autogenerated if missing).
3. **Developer Key Generation (Security-Critical Proposed `@adlc/runner` Extension):** Runs the proposed `adlc-runner init-developer-keys` extension to generate a persistent developer private key pair at `~/.config/opencode/keys/developer.key`. This key is used to sign triage upgrades and manifest evidence. The initialization tool `adlc-runner init-developer-keys` automatically enforces and writes these keys with secure, restricted POSIX permissions (`0600` for private keys, `0644` for public keys) at creation time. If this proposed runner extension is missing or fails, the `/adlc-init` command automatically falls back to generating Ed25519 key pairs using Node's native `crypto` module, writing them with the same secure `0600` permissions and fingerprinting options. The runner checks permissions on startup, refusing to execute signed commands and failing closed if private key permissions are wider than owner-only (`0600`/`0400`). Losing this key prevents cross-session manifest signature verification, requiring a cryptographically signed admin manual override. To generate the admin override, the developer signs a JWT challenge token using an admin private key stored out-of-band in `~/.config/opencode/keys/admin.override` (isolated from the repository). The signature is verified against a frozen admin public key fingerprint (stored in `.git/config` or base repository history). In a single-developer setup, this key acts as a secure, offline recovery mechanism, verified against the frozen public key fingerprint, with compromised keys checked against a local/global revocation list `~/.config/opencode/revocation_blocklist`. To protect against compromised developer keys, a `revokedKeys` fingerprint revocation blocklist can be declared in `.adlc/config.json`, which `adlc-runner` checks to reject signatures from revoked keys.
4. **CI/CD Workflow Scaffolding:** Copies template workflows (`ci/rails-guard.yml` and `ci/adlc-maintenance.yml`) to `.github/workflows/`. If these files already exist, `/adlc-init` will prompt you before overwriting, allowing manual merging.
5. **Hook Installation:** Installs a local pre-commit hook (`.git/hooks/pre-commit`) for environments that lack CI runners, parsing existing hooks to ensure it does not clobber them and runs its validation first.

### Headless/CI and Non-Interactive Triage
For headless or non-interactive environments (such as CI/CD pipelines), the plugin supports automated triage mapping via the `ADLC_TRIAGE_CLASS` environment variable. You can set it to `Trivial`, `Bounded`, `Substantial`, or `Architectural` to bypass the TUI triage interview. If unset, it defaults to `Substantial` (which enforces full specification and decomposition gates, which may fail/deadlock in headless environments if spec inputs are missing). **Precedence rule:** To prevent bypass reward-hacking, the `ADLC_TRIAGE_CLASS` environment variable and `--class` flags only apply during initial triage creation when `.adlc/tickets.json` is missing. Once the triage classification is recorded, any subsequent environment override or downgrading attempts are strictly ignored, and the plugin compares active parameters; any deviation triggers a config mismatch warning and fails closed to prevent downgrade attacks.

---

### 4. Configure LLM Provider API Keys & Frontier Models
Most ADLC gates execute keylessly using OpenCode's active session model. However, to ensure safety boundaries:
1. **Frontier Model Requirement for P1 and P2:** Consistent with ADLC.md, Phase 1 (Interrogate) spec shaping and Phase 2 (Decompose) ticket partitioning require a frontier-tier (Opus-class) model (such as an active frontier-class model) to prevent underspecification and guarantee contract desk stability. Mid-tier (Sonnet-class) and cheap (Haiku-class) models are suitable for Phase 5 (Prosecution) or Phase 4 (Build), but do not cost-optimize Phase 1 or Phase 2 by routing them to mid-tier or cheap models.
2. **Calibration API Keys:** Phase 5 prosecution requires dynamic benchmark calibration (`adlc review-calibration`) which must execute directly against the LLM provider. You must export the corresponding API key (e.g. `ANTHROPIC_API_KEY`, `GEMINI_API_KEY`) in your environment. Failure to configure these keys will cause prosecution to fall back to safe default settings (N = 3 passes, majority k = 2 consensus) and write the "UNCALIBRATED" sentinel, raising a warning notification instead of crashing (unless the developer has explicitly configured a remote frontier model for prosecution/calibration without its API key, which will cause a startup crash).

| `/adlc-rail-write` | P3 | Invokes the `rail-writer` agent to write tests and stubs in an isolated context. Runs `adlc-runner run p3` to assert rail evidence exists and executes a compiler typecheck on the stubs and tests. |
| `/adlc-consensus-fix` | P4 | Runs consensus repair by fanning out candidate fixes to resolve a hard failing test (`adlc consensus-fix`). |
| `/adlc-verify-build` | P4 | Deterministically verifies build gate G4 (asserting compiled code, lint cleanliness, passing rail tests, and that frozen rails remain intact and unmodified with no new suppressions) and runs `adlc-runner run p4` to assert gate completion before prosecution. |
| `/adlc-prosecute` | P5 | Verifies the build gate evidence exists via `adlc-runner run p4` (Proposed `@adlc/runner` Extension; if unavailable, falls back to direct exit checks of the linter and test suite). Triggers `adlc review-calibration` (which runs synchronously, blocking prosecution to prevent race conditions, utilizing direct provider environment keys rather than keyless cascade stdio) on model change or cache miss ([ADLC-C8]). **Trivial run runs a single pass of full 5-lens prosecution** (correctness, security, contract, diff, tests) without looping (routed to mid-tier calibrated models to conform to ADLC D4), and executes `adlc-runner run p5` to assert evidence on success. Runs `adlc-runner run p5` to assert evidence. If findings are detected, it continues the execution of the P4 builder agent in its existing context (retaining the builder conversation history and context, conforming to ADLC Appendix F where fixers continue) and appends the structured findings checklist to guide the repair (or suspends the session and prompts the developer to resume the builder manually if context continuation is unsupported by the harness). |
| `/adlc-accept` | P6 | Finalizes the Phase 6 human gate Human Gate 2 (behavioral acceptance). The command first executes the initial P5 prosecution check to verify safety (which is skipped if valid, signed P5 prosecution evidence matching the active commit hash already exists in the manifest and no files have changed since, preventing redundant token runs); on success, it runs `adlc behavior-diff compare` (which dynamically checks out and runs against the triageCommit baseline inside a temporary git worktree or separate directory clone to prevent clobbering the active workspace and ensure zero data loss risk to current work; if `triageCommit` is missing or rebased out of history, behavior-diff uses a three-step fallback chain: (1) merge-base of the active branch against the default tracking branch, or (2) comparison against the repository's initial commit if merge-base resolution fails, e.g. if no tracking branch is configured for the current branch, or (3) prompting for manual commit hash override) to gather evidence, prompts the developer to verify the demo, and signs the manifest (`adlc-runner accept --ticket <id>`). Both Bounded and Trivial tickets also route through P6, requiring the developer to execute `/adlc-accept` to sign the change before merge. |
| `/adlc-distill` | P7 | Mines findings (`adlc lesson-foundry`, `adlc rejection-mining`) and runs `adlc-runner run p7` to assert distillation evidence. Post-merge Simplify runs in CI; local `/adlc-distill --simplify` runs locally on-demand, temporarily re-locking and freezing the target completed ticket's rail paths during execution to preserve ADLC P7 invariants. |
| `/adlc-maintain` | C10/C12 | Run decay-driven checks locally: stale skills and model-ratchet (`adlc skill-rot` and `adlc model-ratchet`). Note that `skill-rot` runs checks mechanically where possible, and falls back to cheap-model prompts for claims requiring semantic validation (complying with ADLC C10). Running `model-ratchet` is a full prosecution loop over main's hot paths using newest models, requiring direct API keys and incurring inference token costs; the local TUI slash command executes a limited mock dry-run or is skipped entirely unless direct provider keys are supplied, while the full scheduled run is offloaded to CI (`ci/adlc-maintenance.yml`). `adlc gate-fuzzing` runs strictly in CI/CD or VM sandboxed environments to prevent unsandboxed code execution on dev hosts. |

### The Router Skill (Discovery Skill)

The `adlc` router skill acts as a native phase-routing flowchart. Instead of prompting the model to remember 19 CLI tools, the router identifies your current activity ("shape this spec", "verify this bugfix") and instructs OpenCode to reach for the correct ADLC tool or slash command.

---

### The Prosecutor Subagents (Phase 5)

Pre-merge prosecution fanned out across five independent, fresh-context, single-lens subagents to prevent context rot. Consistent with ADLC D4, these subagents run on mid-tier, calibrated models by default, only adding frontier lenses if calibration benchmarks show a category blind spot:
1. **`correctness`**: Off-by-one errors, error swallowing, concurrency, and logic races.
2. **`security`**: Threat modeling, injection vectors, hardcoded secrets, and unsafe input sanitization.
3. **`contract`**: Module boundary validation and interface contract matching. *For Trivial tickets lacking formal decomposition contracts, this lens validates the modified files against the existing type exports and import relationships of the codebase to prevent unauthorized module boundary crossings.*
4. **`diff`**: Compares implementation against the spec to verify all criteria are met. *For Trivial tickets where no specification is synthesised, this lens compares the implementation changes directly against the ticket description and requirements recorded in `.adlc/tickets.json`.*
5. **`tests`**: Audits unit tests added by the builder during the build phase (ensuring they assert behavior instead of mocking reality).

Findings are verified by a separate `prosecutor-verifier` reproducer agent. The loop executes until two consecutive passes run completely dry (zero verified findings) to certify coverage, up to the maximum convergence budget $R_{\text{max}}$ read dynamically from the workspace configuration (`.adlc/config.json`). If the budget is exhausted without drying out, the loop terminates and escalates to the developer to revise the specification or implementation partition.

### In-Process Hooks

OpenCode runs the plugin in-process inside its Bun JavaScript engine, calling the hook checker helper pre-bundled index `plugin/adlc-opencode/dist/index.js` (which packages the rails adapter wrapping `@adlc/core`) on specific event triggers:

| Hook Event | Trigger Event | Posture | Behavior |
| --- | --- | --- | --- |
| **preflight** | `session.created` / `session.start` (including reconnect and resume events) | Advisory | Warns the user if Node/Bun runtimes, git trees, or providers are misconfigured (environment determinism check, [ADLC Appendix F / D2]). |
| **rails-guard** | `tool.execute.before` | **Enforcing** (Requires SDK support; else advisory + CI backstop) | Intercepts structured editing tools using default-intercept (gates all tools with path parameters, excluding safe read-only tools like `read`, `view`, `glob`, `search`, `grep`, or custom read tools declared in `.opencode/opencode.json`'s `allowedReadTools`). The tool `apply_patch` is explicitly classified as a structured edit tool, not a shell tool, and is fully gated by the hook. Denies edits to paths declared as `rails`. During build sessions, pending rails are blocked entirely; pending rails can only be written during a dedicated `/adlc-rail-write` invocation state to defend creator/critic isolation ([ADLC Principle 5]). Once Phase 3 is asserted, rails transition to Frozen (fully blocked). Locks `.adlc/`, `.opencode/`, `.git/`, `.github/`, and `tickets.json` files, parses patches, and scans for unauthorized skips/suppressions (including test.skip, it.skip, describe.skip, it.only, describe.only, test.only, fdescribe, fit, xit, xdescribe, skip/xfail decorators, eslint-disable, ts-ignore, // @ts-expect-error, @ts-nocheck, biome-ignore markers, # type: ignore and # noqa in Python, #[allow(...)] in Rust, Goodhart defense). Hook strictly returns structured block payloads `{ allow: false, reason: "..." }` rather than throwing exceptions to ensure deterministic gating. |
| **flail-detection** | `tool.execute.after` | **Enforcing (with SDK support) / Advisory (Degraded Fallback)** | (Proposed SDK Extension Specification). Scans outputs for flailing. Strike 1: terminates builder subagent, stashes uncommitted changes safely (scoped using `git stash push --include-untracked -- <ticket.domains>...` to target only the active ticket's declared file domain, protecting unrelated changes; for Trivial tickets lacking declared domains, the stash scope falls back to the files changed since `triageCommit` using safe programmatic diff list gathering via `Bun.spawnSync(['git', 'diff', '--name-only', triageCommit])` and stashing via `Bun.spawnSync(['git', 'stash', 'push', '--include-untracked', '--', ...files])` to isolate changes and prevent shell injection, which are expected to be within the 3-file limit under the PreToolUse hook, but will capture any bypassed changes if terminal tools were used, with excess files caught at assertion time), appends logs to `.adlc/tickets.json` history, and restarts builder. Strike 2: terminates builder, stashes staged work, and upgrades the ticket's triage class to Substantial in `.adlc/tickets.json`. The plugin mechanically reverts the active workspace changes (restoring both worktree and index via `git restore --staged --worktree` to clear staged edits) and halts builder agent execution, unconditionally escalating/restarting execution at Phase 2 (Decompose) for all triage classes as mandated by ADLC C6. If the ticket was previously Trivial/Bounded, it upgrades the classification to Substantial in `.adlc/tickets.json` and synthesizes an initial spec under Phase 1 (Interrogate) to maintain Substantial spec requirements (which represents a documented operational deviation from verbatim ADLC C6, as Trivial/Bounded tickets lack a pre-existing spec to run contract verification against, requiring spec synthesis to generate the contract base before Phase 2 decomposition can run), before running Phase 2 decomposition; if it was already Substantial/Architectural, it directly re-decomposes under the upgraded constraints. The upgrade event is signed by `adlc-runner upgrade` using `~/.config/opencode/keys/developer.key` (with permissions restricted to `0600`) and appended to `.adlc/manifest.jsonl` to invalidate prior Trivial records and ensure consistent verification. Note: If the harness SDK lacks suspension APIs, flail-detection degrades to advisory, warning the user but leaving the temporal isolation boundary unprotected. |
| **manifest-audit** | `session.ended` / `session.terminated` | Advisory (Local) / Enforced (CI/CD) | Runs `adlc gate-manifest verify --in-progress` to validate the chronological integrity of records written so far (partial chain validation, e.g. P0, P1, P2) and allow clean session exit, or `adlc gate-manifest verify --complete` to enforce full validation during pre-commit, prosecution, or merge integration. If config or manifest files are deleted/absent but git history shows modifications, or if one of the files is present while the other is absent (half-deletion state), it fails closed (exit code 2). On Trivial tickets, the auto-prosecution check (5 lenses) allows a single builder fix loop (up to $R_{\text{trivial\_fix}} = 2$ rounds, configurable via `maxRoundsTrivial` in `.adlc/config.json`) before upgrading to Substantial. If out-of-session prosecution fails (e.g., during pre-commit or session-end checks), the verification fails closed, stashes files programmatically (`Bun.spawnSync(['git', 'stash', 'push', '-m', 'adlc_trivial_failed_prosecution'])` to prevent shell injection), updates the ticket state, and prompts the developer for builder fixing or upgrade upon reconnection. Out-of-session validation (such as pre-commit hooks or session-end prompts) allows developers to run `adlc behavior-diff compare` in the terminal to review the behavioral diff, and then execute `adlc-runner accept --ticket <id>` to record the P6 signature in the manifest (ensuring behavior evidence is captured before signing). To prevent bypassing Trivial ticket prosecution via force-quit or hook deletion, the preflight hook (on session reconnect/resume) blocks all editing tools and locks the session if the active Trivial ticket has unprosecuted modifications, forcing the execution of `/adlc-prosecute` (or `/adlc-accept`) before any further workspace modifications are permitted or new tickets processed. |

#### Rail Gating Safety & Bypasses
- **Fail-Closed Guarantee:** If a ticket declares rails, but the plugin encounters an operational error, the hook **fails closed (blocks editing)**. If no rails are declared, the hook is a no-op. *Note on onFailure Hook Posture:* If the harness does not support a native `onFailure: deny` hook configuration, any plugin loading error, syntax exception, or startup crash will default-allow (fail open) the in-session hook. In such environments, the rails-guard safety guarantee relies entirely on the CI/CD rails-guard workflow backstop.
- **Pending vs. Frozen Rails:** To enforce creator/critic isolation in environments lacking native agent-identity verification APIs (preventing builder agents from writing their own tests), the hook **blocks all rail edits during builder sessions entirely**. Pending rails can only be written during a dedicated `/adlc-rail-write` invocation state. To prevent a fail-open state where a crash or force-quit during `/adlc-rail-write` leaves the temporal token active permanently, the preflight cleanup routine checks and invalidates stale tokens on `session.created` and `session.start` (including reconnect/resume events) by verifying the status of the platform-agnostic file lock (`.adlc/rail-write.lock` via `proper-lockfile`). If the lock is stale (the lock file exists but is no longer actively held by any running process), the hook resolves this stale state atomically to prevent TOCTOU races between concurrent sessions: it first acquires an exclusive file lock on `.adlc/tickets.lock` (using `proper-lockfile` to serialize concurrent preflight hooks), re-verifies the staleness of `.adlc/rail-write.lock`, and only then invalidates and clears the stale session token from `.adlc/tickets.json`, releasing the tickets lock afterward and restoring the strict rail-guard posture. To prevent temporal bypasses, the plugin TUI suspends active builder session execution while `/adlc-rail-write` is running. Once Phase 3 is complete, rails are marked Frozen and blocked for all agents.
- **Isolated Prompting & Stdio Cascading:** Prompts from LLM-backed gates are queried in isolated sub-contexts. Multi-sample, multi-round prompts bubble up keylessly via the structured Two-Phase Stdio JSON Cascade Protocol (`--prompt-session` flag). Under the `--prompt-only` fallback, Phase 2 re-spawns the tool and pipes the gathered completions back to stdin, ensuring all consensus aggregation remains in the deterministic tool. The bridge enforces strict escaping, sanitization, and verification of model text outputs before embedding them inside the JSON stdin response envelopes to prevent prompt-injection attacks. Verdict records explicitly attach the active review-calibration recall score ([ADLC-C8]) to preserve the C8↔C11 provenance chain.
- **Developer Key & Behavior-Diff Configuration:** Cryptographic signatures for upgrades and all manifest evidence records (both human gate signatures and automated execution logs) are signed using a developer key stored at `~/.config/opencode/keys/developer.key` (enforcing POSIX `0600`/`0400` permissions and UID verification). Every manifest record is cryptographically signed using the developer's key via `adlc-runner record --entry <payload>` at tool completion to verify provenance and maintain a tamper-evident hash-chained append-only evidence chain where each entry links to the hash of the preceding one. Key compromise requires a `revokedKeys` fingerprint revocation blocklist in `.adlc/config.json`, which is verified by `adlc-runner` to reject signatures from revoked keys. Observable behavior validation in Phase 6 runs `behavior-diff compare` using commands and fixture locations configured under the `behaviorDiff` block in `.adlc/config.json` (such as `captureCommand` and `fixturesDir`). If unspecified, it runs the default `testCommand` suite, logging an advisory warning to define explicit behaviors.
- **Frontier-Free Scaling:** Local models that fail dynamic calibrations are scaled up to N-pass parallel checks (ADLC Exploit E1 sampling diversity) and require majority consensus thresholds ($k$-out-of-$N$), dynamically benchmarked during Phase 5 initialization. Calibration (`review-calibration`) runs synchronously (blocking prosecution to prevent race conditions and preserve the C8↔C11 provenance chain) utilizing direct provider environment keys rather than keyless cascade stdio.
- **Absolute Rail Protection (No Bypass):** Conforming to ADLC Principle 5, there is **no bypass mechanism** (such as `ADLC_RAILS_BYPASS=1`) permitted for active rails. If a developer needs to modify rails, they must update the ticket contract or upgrade the ticket class. The hook checker strictly denies all edits to protected paths unconditionally, preventing reward-hacking agents from attempting to bypass enforcement.
- **CI/CD / Local Backstop:** Shell-based rail mutations are blocked at commit-time via GitHub workflows or the local pre-commit hook (configured during `/adlc-init`). The workflow template files for GitHub (`ci/rails-guard.yml` and `ci/adlc-maintenance.yml`) are delivered in the final delivery phase (scaffolded automatically inside the `.github/workflows/` directory during `/adlc-init` execution, or they can be copied manually from the local repository clone's templates directory at `plugins/adlc-opencode/templates/`). The local hook avoids clobbering by parsing the existing file. To support non-shell interpreters (e.g., Python, Perl), `/adlc-init` checks the shebang line; if a non-shell interpreter is detected, it does not inject shell code and instead logs a warning or prompts the user for manual integration rather than corrupting the hook. For shell-based hooks, it identifies any early `exit 0` statements and injects the `adlc rails-guard` validation at the top of the file rather than blindly appending it. The local hook reads the union of rail declarations from `HEAD` and the staged index to ensure local new-rail locking. The CI rails-guard workflow enforces a union strategy (reading rail declarations from both the trunk base ref and head branch) to be deletion-proof against existing base rails while remaining new-rail-aware for newly declared rails in the same PR. To prevent bash-based tampering with rail definitions in `.adlc/tickets.json`, the CI check treats `.adlc/tickets.json` as a frozen rail and rejects deletion/modification of active rails relative to their initial triage commit (identified by reading the baseline triage commit hash stored as `triageCommit` in `.adlc/tickets.json`, failing closed if it is missing, unless the base branch contains no `.adlc/tickets.json` file at all, which represents the repository's first-ever ticket PR and is exempt from base-comparison checks). Furthermore, the `adlc gate-fuzzing` tool verifies sandbox namespaces at execution time, refusing to run outside a container/sandbox. Local hooks are best-effort and must be backstopped by branch protection in CI/CD. Local-only developers must manually inspect hook logs and run direct path diff comparisons against the base branch to verify rail compliance before merging.

#### Local-Only / Non-CI Security Degradation (Warning)

> [!WARNING]
> In workspaces configured without CI/CD branch protection pipelines, the safety contract decays due to the composition of the following six limitations:
> 1. **In-Session Bash Exclusion:** Bash shell commands are not gated in-session, allowing edits to rails via standard terminal tools.
> 2. **Harness Fail-Open Exceptions:** If the OpenCode harness lacks native `onFailure: deny` support, plug-in loading errors or crashes fail open.
> 3. **Agent-Deletable Hooks:** Local pre-commit git hooks are best-effort and can be bypassed or deleted by the agent.
> 4. **No Remote Backstop:** Repositories without CI have no secondary trunk-base diff validation to catch bypassed edits.
> 5. **Temporal Isolation Bypass:** If the proposed TUI suspension API is absent, builder agents cannot be suspended during `/adlc-rail-write` execution, allowing concurrent builder agents to write to pending rails during this temporal window.
> 6. **Evidence Tampering via Bash:** Local evidence files in `.adlc/` are writable via the terminal, enabling agents to falsify test/build logs locally.
>
> Under this composite, there is **zero complete or guaranteed mechanical rail enforcement** during development. Local-only developers must manually audit git logs for edits to `.adlc/` and run direct path comparisons against the base branch to verify rail compliance before merging.
