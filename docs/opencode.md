# Adopt the ADLC in OpenCode

The `@adlc/*` toolkit is a set of gate-shaped CLIs. This plugin makes the whole **Agentic Development Lifecycle** usable natively from inside OpenCode: the gates fire at the right lifecycle moments — some automatically — and the model reaches for the right gate without you memorizing 19 separate tools.

Following **Option D (separate, concern-focused bins)** from the Command Reconciliation ADR:
- `adlc` is the dispatcher (`@adlc/cli`) for executing individual gate tools.
- `adlc-runner` is the runner (`@adlc/runner`) for phase-evidence assertions.

> Design and rationale: [`opencode-integration-plan.md`](./opencode-integration-plan.md).
> The full thesis: [`../ADLC.md`](../ADLC.md).

## Install

### 1. Install the ADLC CLI Tools
The plugin needs both the ADLC dispatcher (`adlc`) and runner (`adlc-runner`) binaries globally resolvable in your PATH.

> [!IMPORTANT]
> Because `@adlc/runner` is a pending-merge dependency from the sibling Codex branch, the convenience bundle `@adlc/all` is not yet published to the public registry. You must install the tools locally:
> 1. Clone the Codex branch of the `adlc` repository.
> 2. Build the binaries locally following the guide in [RELEASING.md](./RELEASING.md).
> 3. Link them to your global path (including the dispatcher, runner, and the 19 independent gate packages, ensuring no first-use ENOENT errors occur):
>    ```sh
>    npm link packages/*
>    ```
> Do NOT mock or stub `adlc-runner` in your PATH; doing so silently disables all phase assertions, leaving the workspace completely unprotected.

The plugin initialization check will verify that both `adlc` and `adlc-runner` bins are available on PATH and output a clear installation error if missing.

### 2. Install the OpenCode Plugin
> [!NOTE]
> The plugin manager command (`opencode plugin add`) and configuration file format (`opencode.json` plugin registry mapping) described below are target specifications that must be verified against the official OpenCode SDK specification during implementation.

You can install the plugin either by adding the published npm package `@adlc/opencode-plugin` directly via OpenCode's plugin manager:

```sh
opencode plugin add @adlc/opencode-plugin
```

Alternatively, for local development, copy or symlink the plugin source directory (`plugin/adlc-opencode`) to your project-local `.opencode/` directory:

```sh
mkdir -p .opencode/plugin/
# Copy or symlink the integration code:
ln -s /path/to/adlc/plugin/adlc-opencode/ .opencode/plugin/adlc-opencode
```

Alternatively, register it globally in your `~/.config/opencode/opencode.json`:

```json
{
  "plugin": [
    "/path/to/adlc/plugin/adlc-opencode"
  ]
}
```

### 3. Initialize your Repository
Bootstrap the ADLC workspace context in your repository by typing the following slash command directly into the OpenCode TUI chat interface:

```text
/adlc-init
```

This creates the `.adlc/` workspace configuration directory, templates the `.adlc/config.json` and `.adlc/tickets.json` structures, appends runtime logs and temp directories idempotently to your `.gitignore`, scaffolds the local `.opencode/` subdirectories (`command/`, `agent/`, `skill/`) and configuration `opencode.json` (autogenerated if missing), and sets up a local pre-commit hook (`.git/hooks/pre-commit`) for environments that lack CI runners.

### 4. Configure LLM Provider API Keys & Frontier Models
Most ADLC gates execute keylessly using OpenCode's active session model. However, to ensure safety boundaries:
1. **Frontier Model Requirement for P1 and P2:** Consistent with ADLC.md, Phase 1 (Interrogate) spec shaping and Phase 2 (Decompose) ticket partitioning require a frontier model (such as Claude 3 Opus or Gemini 1.0 Ultra / 1.5 Pro where Ultra/Opus-class is configured in `.adlc/config.json`, following the Opus-class ceiling assumption of ADLC.md) to prevent underspecification and guarantee contract desk stability. Do not cost-optimize these phases.
2. **Calibration API Keys:** Phase 5 prosecution requires dynamic benchmark calibration (`adlc review-calibration`) which must execute directly against the LLM provider. You must export the corresponding API key (e.g. `ANTHROPIC_API_KEY`, `GEMINI_API_KEY`) in your environment. Failure to configure these keys will cause prosecution to crash at Phase 5 startup.

## What You Get

### Commands (Slash Commands)

Trigger these directly within the OpenCode TUI interface:

| Command | Phase | Description |
| --- | --- | --- |
| `/adlc-init` | — | Bootstrap `.adlc/` workspace, scaffold `.opencode/` structure (linking/copying commands, agents, and skills from the plugin), template `.adlc/config.json` defaults, configure gitignore (idempotently), configure local pre-commit git hooks, and check environment readiness. |
| `/adlc-ticket` | P0 | Runs an interactive triage interview to evaluate risk × blast radius (or parses non-interactively in headless runs via arguments/environment variables) and authors a ticket contract. Dispatcher (`adlc`) mechanical blast-radius overrides act as the primary unbypassable F5-defense control (upgrading Trivial tickets automatically if target tests are missing, hollow, or if the diff exceeds limits); `adlc-runner run p0` (Proposed `@adlc/runner` Extension) asserts triage evidence. **Note:** /adlc-ticket executes a quick pre-triage test suite and hollow-test execution to verify pre-existing coverage, which introduces latency depending on test suite size. Trivial ticket builders run strictly on cheap/low-tier models to minimize execution cost. |
| `/adlc-spec` | P1 | Runs the spec interrogation flow. The command executes a structured "grillme" Q&A interview to extract requirements from the developer's head, requiring a frontier model configuration. If the ticket class is Architectural, it runs `adlc parallax` to generate alternative spec drafts. Once a spec draft is established, it runs `adlc spec-lint` to validate it and `adlc premortem` to stress-test it, ensuring a coherent build-start flow. |
| `/adlc-approve-spec` | P1 | Finalizes the Phase 1 human gate Human Gate 1 (spec approval). Signs the manifest and records the spec approval hash. `adlc-runner run p1` is then run to assert both spec audit evidence and Human Gate 1 signature exist. |
| `/adlc-decompose` | P2 | Runs `adlc coldstart`, `adlc model-router`, and `adlc merge-forecast` to split the ticket and verify contract boundaries. Runs `adlc-runner run p2` to assert evidence. |
| `/adlc-rail-write` | P3 | Invokes the `rail-writer` agent to write tests and stubs in an isolated context, and runs `adlc hollow-test` to verify tests fail on implementation deletion before freeze (locked to the `rail-writer` agent context to defend creator/critic isolation; if harness APIs prevent dynamic agent identity verification in-session, validation defaults to `hollow-test` + the temporal session-token mechanism and builder agent suspension). Runs `adlc-runner run p3` to assert rail evidence exists. |
| `/adlc-consensus-fix` | P4 | Runs consensus repair by fanning out candidate fixes to resolve a hard failing test (`adlc consensus-fix`). |
| `/adlc-verify-build` | P4 | Deterministically verifies build gate G4 (asserting compiled code, lint cleanliness, passing rail tests, and that frozen rails remain intact and unmodified with no new suppressions) and runs `adlc-runner run p4` to assert gate completion before prosecution. |
| `/adlc-prosecute` | P5 | Verifies the build gate evidence exists via `adlc-runner run p4` (Proposed `@adlc/runner` Extension). Triggers `adlc review-calibration` (which runs synchronously, blocking prosecution to prevent race conditions, utilizing direct provider environment keys rather than keyless cascade stdio) on model change or cache miss ([ADLC-C8]). **Trivial run runs a single pass of full 5-lens prosecution** (correctness, security, contract, diff, tests) without looping, and executes `adlc-runner run p5` to assert evidence on success. **Bounded run runs prosecute-lite**: 2 lenses (correctness and diff) looping until 2 consecutive passes run dry to comply with the F6 loop-until-dry defense. **Substantial/Architectural runs full prosecution**: 5 lenses fanned out and looping until 2 consecutive passes run dry (zero findings), up to $R_{\text{max}}$ rounds (blocking integration and failing closed if convergence is not reached). Runs `adlc-runner run p5` to assert evidence. If findings are detected, it continues the execution of the P4 builder agent in its existing context (retaining the builder conversation history and context, conforming to ADLC Appendix F where fixers continue) and appends the structured findings checklist to guide the repair (or suspends the session and prompts the developer to resume the builder manually if context continuation is unsupported by the harness). |
| `/adlc-accept` | P6 | Finalizes the Phase 6 human gate Human Gate 2 (behavioral acceptance). Runs `adlc behavior-diff compare` (which dynamically checks out and runs against the triageCommit baseline inside a temporary git worktree or separate directory clone to prevent clobbering the active workspace and ensure zero data loss risk to current work) to gather evidence, prompts the developer to verify the demo, and signs the manifest (`adlc-runner accept --ticket <id>`). Both Bounded and Trivial tickets also route through P6, requiring the developer to execute `/adlc-accept` to sign the change before merge. |
| `/adlc-distill` | P7 | Mines findings (`adlc lesson-foundry`, `adlc rejection-mining`) and runs `adlc-runner run p7` to assert distillation evidence. Post-merge Simplify runs in CI; local `/adlc-distill --simplify` runs locally on-demand, temporarily re-locking and freezing the target completed ticket's rail paths during execution to preserve ADLC P7 invariants. |
| `/adlc-maintain` | C10/C12 | Run safe decay-driven checks locally: stale skills and model-ratchet (`adlc skill-rot` and `adlc model-ratchet`). Note that `skill-rot` runs checks mechanically where possible, and falls back to cheap-model prompts for claims requiring semantic validation (complying with ADLC C10). `adlc gate-fuzzing` runs strictly in CI/CD or VM sandboxed environments to prevent unsandboxed code execution on dev hosts. |

### The Router Skill (Discovery Skill)

The `adlc` router skill acts as a native phase-routing flowchart. Instead of prompting the model to remember 19 CLI tools, the router identifies your current activity ("shape this spec", "verify this bugfix") and instructs OpenCode to reach for the correct ADLC tool or slash command.

---

### The Prosecutor Subagents (Phase 5)

Pre-merge prosecution fanned out across five independent, fresh-context, single-lens subagents to prevent context rot. Consistent with ADLC D4, these subagents run on mid-tier, calibrated models by default, only adding frontier lenses if calibration benchmarks show a category blind spot:
1. **`correctness`**: Off-by-one errors, error swallowing, concurrency, and logic races.
2. **`security`**: Threat modeling, injection vectors, hardcoded secrets, and unsafe input sanitization.
3. **`contract`**: Module boundary validation and interface contract matching.
4. **`diff`**: Compares implementation against the spec to verify all criteria are met.
5. **`tests`**: Audits unit tests added by the builder during the build phase (ensuring they assert behavior instead of mocking reality).

Findings are verified by a separate `prosecutor-verifier` reproducer agent. The loop executes until two consecutive passes run completely dry (zero verified findings) to certify coverage, up to the maximum convergence budget $R_{\text{max}}$ read dynamically from the workspace configuration (`.adlc/config.json`). If the budget is exhausted without drying out, the loop terminates and escalates to the developer to revise the specification or implementation partition.

### In-Process Hooks

OpenCode runs the plugin in-process inside its Bun JavaScript engine, calling the hook checker helper pre-bundled index `plugin/adlc-opencode/dist/index.js` (which packages the rails adapter wrapping `@adlc/core`) on specific event triggers:

| Hook Event | Trigger Event | Posture | Behavior |
| --- | --- | --- | --- |
| **preflight** | `session.created` / `session.start` (including reconnect and resume events) | Advisory | Warns the user if Node/Bun runtimes, git trees, or providers are misconfigured (environment determinism check, [ADLC Appendix F / D2]). |
| **rails-guard** | `tool.execute.before` | **Enforcing** (Requires SDK support; else advisory + CI backstop) | Intercepts structured editing tools using default-intercept (gates all tools with path parameters, excluding safe read-only tools like `read`, `view`, `glob`, `search`, `grep`, or custom read tools declared in `.opencode/opencode.json`'s `allowedReadTools`). Denies edits to paths declared as `rails`. During build sessions, pending rails are blocked entirely; pending rails can only be written during a dedicated `/adlc-rail-write` invocation state to defend creator/critic isolation ([ADLC Principle 5]). Once Phase 3 is asserted, rails transition to Frozen (fully blocked). Locks `.adlc/`, `.opencode/`, `.git/`, `.github/`, and `tickets.json` files, parses patches, and scans for unauthorized skips/suppressions (including test.skip, it.skip, describe.skip, it.only, describe.only, test.only, fdescribe, fit, xit, xdescribe, skip/xfail decorators, eslint-disable, ts-ignore, @ts-nocheck, and biome-ignore markers, Goodhart defense). Hook strictly returns structured block payloads `{ allow: false, reason: "..." }` rather than throwing exceptions to ensure deterministic gating. |
| **flail-detection** | `tool.execute.after` | **Enforcing (with SDK support) / Advisory (Degraded Fallback)** | (Proposed SDK Extension Specification). Scans outputs for flailing. Strike 1: terminates builder subagent, stashes uncommitted changes safely (scoped using `git stash push --include-untracked -- <ticket.domains>...` to target only the active ticket's declared file domain, protecting unrelated changes; for Trivial tickets lacking declared domains, the stash scope falls back to the files changed since `triageCommit` (using `git stash push --include-untracked`), which are expected to be within the 3-file limit under the PreToolUse hook, but will capture any bypassed changes if terminal tools were used, with excess files caught at assertion time), appends logs to `.adlc/tickets.json` history, and restarts builder. Strike 2: terminates builder, stashes staged work, and upgrades the ticket's triage class to Substantial in `.adlc/tickets.json`. The plugin mechanically reverts the active workspace changes (restoring both worktree and index via `git restore --staged --worktree` to clear staged edits) and halts builder agent execution, forcing the ticket to restart at Phase 1 (Interrogate) if the ticket was previously Trivial/Bounded (to author a proper specification for the upgraded Substantial ticket before decomposition), or Phase 2 (Decompose) if it was already Substantial/Architectural to refine and re-decompose the ticket partitions. The upgrade event is appended to `.adlc/manifest.jsonl` to invalidate prior Trivial records and ensure consistent verification. Note: If the harness SDK lacks suspension APIs, flail-detection degrades to advisory, warning the user but leaving the temporal isolation boundary unprotected. |
| **manifest-audit** | `session.ended` / `session.terminated` | Advisory (Local) / Enforced (CI/CD) | Runs `gate-manifest verify` to confirm the integrity of the append-only evidence chain on session close (which vacuously passes with code 0 on missing files for newly initialized repos where tickets.json is empty or absent and no git commit history modified ticket files exists; if tickets exist or tickets.json is absent/deleted but manifest.jsonl exists or contains records, or if git history shows ticket files were modified, it fails closed with exit code 2 to prevent bypass by deletion). On Trivial tickets, `session.ended`, `session.terminated`, or pre-commit events automatically trigger a single pass of the P5 prosecute check (running all 5 lenses: correctness, security, contract, diff, tests) and run the build gate check prior to execution, executing the runner assertion `adlc-runner run p5` as an automated side effect on success, and upgrading to Substantial if any findings are found. |

#### Rail Gating Safety & Bypasses
- **Fail-Closed Guarantee:** If a ticket declares rails, but the plugin encounters an operational error, the hook **fails closed (blocks editing)**. If no rails are declared, the hook is a no-op. *Note on onFailure Hook Posture:* If the harness does not support a native `onFailure: deny` hook configuration, any plugin loading error, syntax exception, or startup crash will default-allow (fail open) the in-session hook. In such environments, the rails-guard safety guarantee relies entirely on the CI/CD rails-guard workflow backstop.
- **Pending vs. Frozen Rails:** To enforce creator/critic isolation in environments lacking native agent-identity verification APIs (preventing builder agents from writing their own tests), the hook **blocks all rail edits during builder sessions entirely**. Pending rails can only be written during a dedicated `/adlc-rail-write` invocation state. To prevent a fail-open state where a crash or force-quit during `/adlc-rail-write` leaves the temporal token active permanently, the preflight cleanup routine checks and invalidates stale tokens on `session.created` and `session.start` (including reconnect/resume events) by verifying the status of the platform-agnostic file lock (`.adlc/rail-write.lock` via `proper-lockfile`); if the lock is stale (the lock file exists but is no longer actively held by any running process), the hook automatically invalidates and clears the stale session token from `.adlc/tickets.json`, restoring the strict rail-guard posture. To prevent temporal bypasses, the plugin TUI suspends active builder session execution while `/adlc-rail-write` is running. Once Phase 3 is complete, rails are marked Frozen and blocked for all agents.
- **Isolated Prompting & Stdio Cascading:** Prompts from LLM-backed gates are queried in isolated sub-contexts. Multi-sample, multi-round prompts bubble up keylessly via the structured Two-Phase Stdio JSON Cascade Protocol (`--prompt-session` flag). Under the `--prompt-only` fallback, Phase 2 re-spawns the tool and pipes the gathered completions back to stdin, ensuring all consensus aggregation remains in the deterministic tool. The bridge enforces strict escaping, sanitization, and verification of model text outputs before embedding them inside the JSON stdin response envelopes to prevent prompt-injection attacks. Verdict records explicitly attach the active review-calibration recall score ([ADLC-C8]) to preserve the C8↔C11 provenance chain.
- **Frontier-Free Scaling:** Local models that fail dynamic calibrations are scaled up to N-pass parallel checks (ADLC Exploit E1 sampling diversity) and require majority consensus thresholds ($k$-out-of-$N$), dynamically benchmarked during Phase 5 initialization. Calibration (`review-calibration`) runs synchronously (blocking prosecution to prevent race conditions and preserve the C8↔C11 provenance chain) utilizing direct provider environment keys rather than keyless cascade stdio.
- **Bypass hatch:** Setting `ADLC_RAILS_BYPASS=1` overrides the in-session hook, but requires human approval in the TUI and logs the bypass event to `.adlc/manifest.jsonl`. Bypass is refused (fails closed) in headless sessions.
- **CI/CD / Local Backstop:** Shell-based rail mutations are blocked at commit-time via GitHub workflows or the local pre-commit hook (configured during `/adlc-init`). The workflow template files for GitHub (`ci/rails-guard.yml` and `ci/adlc-maintenance.yml`) are delivered in the final delivery phase (scaffolded automatically inside the `.github/workflows/` directory during `/adlc-init` execution, or they can be copied manually from the local repository clone's templates directory at `plugins/adlc-opencode/templates/`). The local hook avoids clobbering by parsing the existing file. To support non-shell interpreters (e.g., Python, Perl), `/adlc-init` checks the shebang line; if a non-shell interpreter is detected, it does not inject shell code and instead logs a warning or prompts the user for manual integration rather than corrupting the hook. For shell-based hooks, it identifies any early `exit 0` statements and injects the `adlc rails-guard` validation at the top of the file rather than blindly appending it. The local hook reads the union of rail declarations from `HEAD` and the staged index to ensure local new-rail locking. The CI rails-guard workflow enforces a union strategy (reading rail declarations from both the trunk base ref and head branch) to be deletion-proof against existing base rails while remaining new-rail-aware for newly declared rails in the same PR. To prevent bash-based tampering with rail definitions in `.adlc/tickets.json`, the CI check treats `.adlc/tickets.json` as a frozen rail and rejects deletion/modification of active rails relative to their initial triage commit (identified by reading the baseline triage commit hash stored as `triageCommit` in `.adlc/tickets.json`, failing closed if it is missing). Furthermore, the `adlc gate-fuzzing` tool verifies sandbox namespaces at execution time, refusing to run outside a container/sandbox. Local hooks are best-effort and must be backstopped by branch protection in CI/CD. Local-only developers must manually inspect hook logs and run direct path diff comparisons against the base branch to verify rail compliance before merging.

#### Local-Only / Non-CI Security Degradation (Warning)

> [!WARNING]
> In workspaces configured without CI/CD branch protection pipelines, the safety contract decays due to the composition of the following six limitations:
> 1. **In-Session Bash Exclusion:** Bash shell commands are not gated in-session, allowing edits to rails via standard terminal tools.
> 2. **Harness Fail-Open Exceptions:** If the OpenCode harness lacks native `onFailure: deny` support, plug-in loading errors or crashes fail open.
> 3. **Agent-Deletable Hooks:** Local pre-commit git hooks are best-effort and can be bypassed or deleted by the agent.
> 4. **No Remote Backstop:** Repositories without CI have no secondary trunk-base diff validation to catch bypassed edits.
> 5. **Temporal Isolation Bypass:** If the proposed TUI suspension API is absent, builder agents cannot be suspended during `/adlc-rail-write` execution, allowing concurrent builder agents to write to pending rails during this temporal window.
> 6. **Evidence Tampering via Bash:** Local evidence files in `.adlc/` are writable via the terminal, enabling agents to falsify test/build logs locally.
>
> Under this composite, there is **zero complete or guaranteed mechanical rail enforcement** during development. Local-only developers must manually audit git logs for edits to `.adlc/` and run direct path comparisons against the base branch to verify rail compliance before merging.
